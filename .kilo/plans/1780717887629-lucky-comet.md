# Plan: TextSourceBuffer — Fake SourceBuffer for DASH Text Tracks

**Goal:** Implement a `TextSourceBuffer` that satisfies the same interface as
`ManagedSourceBuffer`, parses fMP4-wrapped text (WebVTT-in-ISOBMFF `wvtt`,
TTML-in-ISOBMFF `stpp`), injects `VTTCue` objects into a `TextTrack` owned by
the video element, and maintains all SourceBuffer-like semantics (buffered
ranges, remove, timestampOffset, abort, replace-on-append). The rest of the
player tree treats text adaptation sets identically to audio/video — no
special-casing below `videl-player`.

---

## Architecture overview

```
videl-player
  #setupMse()
    → content-type="video"  → ms.addSourceBuffer() → ManagedSourceBuffer
    → content-type="audio"  → ms.addSourceBuffer() → ManagedSourceBuffer
    → content-type="text"   → TextSourceBuffer(videoEl, label, lang)
                              (NO real SourceBuffer created)
    → ads.sourceBuffer = <ISourceBuffer>   (same property, same interface)

videl-adaptation-set → videl-representation → videl-segment
  segment.sourceBuffer.append(bytes)   ← TextSourceBuffer.append() parses + injects cues
  segment.sourceBuffer.remove(s,e)     ← TextSourceBuffer.remove() deletes cues
```

The `TextTrack` is owned by `TextSourceBuffer`, created once via
`videoEl.addTextTrack('subtitles', label, lang)` in the constructor. The track
is hidden (`mode='hidden'`) until the consumer sets it to `'showing'` (the
standard pattern; we expose the raw `TextTrack` reference on the object).

### `src/text-track/fmp4-text-demuxer.ts`

Stateful class that accumulates init-segment data and parses media segments.

```typescript
export interface TextSample {
  pts:      number;   // presentation-timeline seconds (DT/timescale, before timestampOffset)
  duration: number;   // seconds
  data:     Uint8Array;
}

/**
 * Container-level codec, read from the stsd sample-entry fourcc in the init
 * segment. This is always either 'wvtt' or 'stpp' regardless of TTML sub-profile.
 * Sub-profile discrimination is done at the TextSourceBuffer level using the
 * codec string from the MPD manifest.
 */
export type TextContainerCodec = 'wvtt' | 'stpp' | 'unknown';

export class Fmp4TextDemuxer {
  get containerCodec(): TextContainerCodec { … }
  get timescale(): number { … }

  /** Feed the init segment (moov box). Parses timescale + container codec. */
  parseInit(data: ArrayBuffer): void { … }

  /** Feed a media segment (moof + mdat). Returns decoded samples. */
  parseMedia(data: ArrayBuffer): TextSample[] { … }
}
```

**Init-segment parsing (`parseInit`):**
- Walk ISOBMFF boxes: find `moov → trak → mdia → mdhd` → read `timescale`
- Find `moov → trak → mdia → minf → stbl → stsd` → read first sample-entry
  fourcc: `wvtt` or `stpp`

**Media-segment parsing (`parseMedia`):**
- Find `moof → traf → tfdt` → `baseMediaDecodeTime` (64-bit)
- Find `moof → traf → trun` → per-sample `{ sampleDuration, sampleOffset,
  sampleSize }` array; accumulate decode time offsets from `tfdt`
- For each sample: slice the matching byte range from `mdat`; compute
  `pts = (baseMediaDecodeTime + offsetAccum) / timescale`
- Return `TextSample[]`

Box-reading uses a simple iterative walker (no recursion needed beyond two
levels). All box sizes/offsets are read from 32-bit big-endian integers with a
64-bit extended-size path.

### `src/text-track/wvtt-parser.ts`

Parses the payload of a single `wvtt` sample (a sequence of nested ISOBMFF

```typescript
export interface VttCueData {
  id:       string;
  payload:  string;   // raw VTT cue text
  settings: string;   // VTT cue settings string (may be empty)
}

export function parseWvttSample(data: Uint8Array): VttCueData | null { … }
```

Each `wvtt` sample contains a `vttc` box with optional sub-boxes:
- `iden` — cue identifier (string)
- `sttg` — cue settings (string)  
- `payl` — cue payload text (string)

The parser reads these sub-boxes and returns one `VttCueData`. Samples without
a `vttc` box (empty cue / filler box `vtte`) return `null`.

### `src/text-track/stpp-parser.ts`

TTML/IMSC parser for `stpp` samples. Each sample is a UTF-8 TTML XML fragment.

V1 scope: parse `<p>` elements with `begin`/`end` timing attributes using
`DOMParser`, concatenate text content as the cue payload, ignore TTML
regions/styling/animation.

TTML `begin`/`end` timing uses clock-time format (`HH:MM:SS.fraction` or
`HH:MM:SS:frames`), which requires a small dedicated parser.

```typescript
export function parseStppSample(data: Uint8Array): VttCueData[] { … }
```

Uses `DOMParser`.

**This single function covers all text-profile `stpp` variants** because IMSC1
text (`stpp.ttml.im1t`), IMSC2 text (`stpp.ttml.im2t`), and EBU-TT-D
(`stpp.ttml.etd1`) are all strict subsets of TTML that use the same `<p>`
element structure with `begin`/`end` attributes. No per-profile branching is
needed in the parser itself.

### `src/text-track/text-source-buffer.ts`

The main class. Implements `ISourceBuffer`.

```typescript
export class TextSourceBuffer implements ISourceBuffer {
  readonly textTrack: TextTrack;

  /**
   * @param videoEl   The HTMLVideoElement that owns the TextTrack.
   * @param label     Display label (from MPD Label or lang attribute).
   * @param lang      BCP-47 language tag.
   * @param codecHint Full codec string from the MPD manifest
   *                  (e.g. "wvtt", "stpp", "stpp.ttml.im1t").
   *                  Used to gate image-based profiles and guide stpp parsing.
   */
  constructor(
    videoEl:   HTMLVideoElement,
    label:     string,
    lang:      string,
    codecHint: string = '',
  ) {
    this.textTrack = videoEl.addTextTrack('subtitles', label, lang);
    this.textTrack.mode = 'hidden';
    this.#setCodecHint(codecHint);
  }

  // ── ISourceBuffer ──────────────────────────────────────────────────────────

  async append(data: ArrayBuffer | ArrayBufferView): Promise<void> { … }
  async remove(start: number, end: number): Promise<void> { … }
  async abort(): Promise<void> { … }
  /**
   * Update the codec hint (e.g. after an ABR switch to a representation
   * with a different codec string). Resets the demuxer so the next init
   * segment is parsed fresh. For image-profile codecs, sets the unsupported
   * flag so subsequent appends are silently skipped.
   */
  changeType(mimeAndCodecs: string): void { … }
  get updating(): boolean { … }
  get buffered(): TimeRanges { … }
  timestampOffset = 0;
}
```

**Supported codec strings:**

| Codec string | Container codec | Profile | Supported |
|---|---|---|---|
| `wvtt` | `wvtt` | WebVTT-in-ISOBMFF (ISO 14496-30) | ✓ |
| `stpp` | `stpp` | Generic TTML-in-ISOBMFF | ✓ (basic `<p>`) |
| `stpp.ttml.im1t` | `stpp` | IMSC1 text profile | ✓ (subset of TTML, same parser) |
| `stpp.ttml.im1i` | `stpp` | IMSC1 image profile | ✗ — bitmap rendering not supported |
| `stpp.ttml.etd1` | `stpp` | EBU-TT-D | ✓ (same TTML `<p>` parser) |
| `stpp.ttml.im2t` | `stpp` | IMSC2 text profile | ✓ (same TTML `<p>` parser) |
| `stpp.ttml.im2i` | `stpp` | IMSC2 image profile | ✗ — bitmap rendering not supported |

The codec string is parsed via a helper `classifyTextCodec(codecString)`:

```typescript
type TextCodecClass =
  | { kind: 'wvtt' }
  | { kind: 'stpp-text' }          // all text-profile stpp variants
  | { kind: 'stpp-image' }         // image profiles — unsupported
  | { kind: 'unknown' };

function classifyTextCodec(codec: string): TextCodecClass {
  const c = codec.trim().toLowerCase();
  if (c === 'wvtt') return { kind: 'wvtt' };
  if (c === 'stpp.ttml.im1i' || c === 'stpp.ttml.im2i') return { kind: 'stpp-image' };
  if (c.startsWith('stpp')) return { kind: 'stpp-text' };
  return { kind: 'unknown' };
}
```

This lives in `src/text-track/text-codec.ts` (small utility, exported from
`src/index.ts`).

The codec string originates from the MPD parser: the ADS `codecs` attribute
(or the first representation's `codecs` attribute if absent at ADS level) is
passed to `videl-player` which forwards it to `TextSourceBuffer` as `codecHint`.

**Internal queue** — same serialization pattern as `ManagedSourceBuffer`: a
`queue` array of `{ kind, args, resolve, reject }` entries; `#isProcessing`
flag; `#processQueue()` that dequeues one entry and resolves it when done. Text
parsing is synchronous so each operation resolves on the same microtask — no
`updateend` event wait needed.

**`append` implementation:**

1. If the codec class is `stpp-image` or `unknown`: resolve immediately without
   processing (silently discard). Log a `console.warn` once on first skip.
2. Determine if this is an init segment (contains `moov` box fourcc at offset 4)
   or a media segment.
3. For init: call `this.#demuxer.parseInit(data)` — updates timescale/container
   codec; no cues to inject; `#bufferedRanges` is NOT updated for init segments
   (same as a real SourceBuffer — init data has no presentation time range).
4. For media: call `this.#demuxer.parseMedia(data)` → `TextSample[]`.
5. For each sample:
   a. Apply `timestampOffset`: `presentationTime = sample.pts + timestampOffset`
   b. **Replace-on-append**: call `this.#removeCuesInRange(presentationTime,
      presentationTime + sample.duration)` before adding new cues.
   c. Parse the sample payload via `parseWvttSample` (codec class `wvtt`) or
      `parseStppSample` (codec class `stpp-text`) — routing by `#codecClass`,
      NOT by `this.#demuxer.containerCodec`, so that the manifest-level codec
      string always governs dispatch.
   d. For each `VttCueData`: construct a `VTTCue(startTime, endTime, payload)`,
      set `cue.id` and `cue.line`/`cue.position`/`cue.align` from settings,
      call `this.textTrack.addCue(cue)`.
6. Update `#bufferedRanges`: call `#bufferedRanges.add(minPts + timestampOffset,
   maxPts + maxDuration + timestampOffset)`.

**`changeType(mimeAndCodecs)` implementation:**

Parse the codecs parameter from the `mimeAndCodecs` string (same format as
`MediaSource.isTypeSupported`): extract the codec token, call
`classifyTextCodec`, update `#codecClass`, and reset the demuxer so the next
`append` of an init segment is re-parsed clean. This handles the rare case of
an ABR switch between representations with different text codecs (e.g. switching
from `wvtt` to `stpp.ttml.im1t`).

**`remove(start, end)` implementation:**

1. Call `#removeCuesInRange(start, end)` — iterates `textTrack.cues`, removes
   any whose `startTime < end && endTime > start` (overlap check).
2. Call `#bufferedRanges.cut(start, end)`.

**`#removeCuesInRange(start, end)`:** Snapshot `Array.from(textTrack.cues ?? [])`,
filter for overlap, call `textTrack.removeCue(cue)` on each.

**`abort()`:** Clears the pending queue (rejecting queued entries), resolves
when the in-flight op (if any) is done. Mirrors `ManagedSourceBuffer.abort()`.

## Behaviour contract (matching real SourceBuffer semantics)

| Behaviour | `SourceBuffer` | `TextSourceBuffer` |
|---|---|---|
| Init segment append | Parses codec config | Reads timescale + codec type |
| Media segment append | Muxes into buffer | Parses + injects `VTTCue` objects |
| Buffered range tracking | Browser native | `SyntheticTimeRanges` |
| `remove(s,e)` | Evicts media bytes | Removes cues in range; trims buffered |
| Replace-on-append | Browser handles | Explicit: remove existing cues in new segment range before inserting |
| `abort()` | Flushes queue | Flushes queue; resolves in-flight op |
| `timestampOffset` | Applied to DTS | Applied to decoded PTS before cue construction |
| `changeType()` | Codec switch | No-op (text codec is fixed at init) |
| `updating` | True while appending | True while operation in flight |
| `endOfStream` interaction | Gates EOS | Excluded from EOS check |

## TextTrack visibility — "None" adaptation set

Rather than a synthetic row in the period's shadow DOM, a real
`<videl-adaptation-set>` element is injected as the **first** text ADS in each
period, with `content-type="text"` and a `videl-text-none` boolean attribute and a
`label` of `"None"`. It participates in the full slot lifecycle: it is activated
as the default (period's `selectAdaptationSet` returns `candidates[0]`), renders
its own `.track` row in the subtitle menu, and can be clicked to fire
`videl:track:select` like any other ADS. No special-casing in the period's
shadow DOM template is needed.

### Where the "None" ADS is injected

The **MPD parser** (`mpd-parser.ts`) injects the "None" ADS. After building the
`videl-adaptation-set` elements for a given period, if any have
`content-type="text"`, a "None" ADS is prepended to the text group:

```typescript
const noneAds = document.createElement('videl-adaptation-set');
noneAds.setAttribute('content-type', 'text');
noneAds.setAttribute('videl-text-none', '');
noneAds.setAttribute('label', 'None');
periodEl.insertBefore(noneAds, firstTextAds);
```

The "None" ADS has no `<videl-representation>` children and no
`initialization-url` — it is intentionally inert as a data source. When
activated, it simply hides the `TextTrack`; it never fetches or appends
anything.

### ISourceBuffer — optional visibility hooks

Add two optional methods to `ISourceBuffer`:

```typescript
show?(): void;   // called when a real text track becomes active
hide?(): void;   // called when the "None" track becomes active
```

`ManagedSourceBuffer` does not implement these (undefined → callers use `?.`
optional chaining; no-op for audio/video). `TextSourceBuffer` implements them:

```typescript
show(): void { this.textTrack.mode = 'showing'; }
hide(): void { this.textTrack.mode = 'hidden';  }
```

### show/hide wired in videl-adaptation-set.attributeChangedCallback

`videl-adaptation-set` gains a `videlTextNone` reactive property
(`attribute: 'videl-text-none'`, boolean). In `attributeChangedCallback`, when
`videl-state` becomes `'active'`:

```typescript

  if (this.videlTextNone) {
    this.#sourceBuffer?.hide?.();
  } else {
    this.#sourceBuffer?.show?.();

```

This keeps the show/hide entirely within the ADS, with zero changes to the
period or player beyond removing the text-skip guard.

`#onTrackClick` in `videl-adaptation-set` currently bails for non-audio:

Extend to also allow text (including the "None" ADS):

### Changes to videl-period — #onTrackSelect handles text too

Remove the audio-only guard and generalise to audio + text:

For **text** switches (including switching to "None"):
- Trim the source buffer: `sb.remove(this.#lastCurrentTime, Infinity)` — clears

## presentationTimeOffset wiring (applies to all content types)

`@presentationTimeOffset` (parsed as `pto` in the internal `SegTemplate` type)
tells the decoder how far the media decode timeline is offset from the
presentation timeline. The formula for converting a raw fMP4 decode time (DT,
in timescale ticks) to a presentation time is:

```
presentationTime = periodStart + (DT - pto) / timescale
                 = DT / timescale + (periodStart - pto / timescale)
                 = DT / timescale + timestampOffset
```

where `timestampOffset = periodStart - pto / timescale`.

For **real SourceBuffers** (audio/video), the browser applies
`SourceBuffer.timestampOffset` automatically. This property is currently **not
set** by the player — it works for test streams where `pto = 0` but is wrong
for any stream with a non-zero `@presentationTimeOffset`.

For **TextSourceBuffer**, the time math is done in user code so we must apply
it explicitly.

The fix is applied at three layers:

### 1. Parser — stamp `timestamp-offset` on videl-representation

In `buildSegments` (after `st` is resolved via `mergeSegTemplate`), stamp the
computed value on `repEl` before building the segment list:

```typescript
// In buildSegments(), after: const { base, st, … } = ctx;
if (st) {
  const tsOffset = ctx.periodStart - (st.pto ?? 0) / (st.timescale ?? 1);
  if (tsOffset !== 0) {
    repEl.setAttribute('timestamp-offset', String(tsOffset));
  }
}
```

This covers all segment addressing paths (SegmentTimeline, SegmentTemplate with
`@duration`, SegmentList, SegmentBase). The attribute is omitted when zero to
keep the DOM uncluttered for the common case.

### 2. videl-representation — set timestampOffset on sourceBuffer

Add a `timestampOffset` reactive property (`attribute: 'timestamp-offset'`,
type `Number`, default `0`). In `#startInit()`, after the init segment is
successfully appended, set it on the source buffer:

```typescript
// After: this.#initAppended = true;
if (this.#sourceBuffer && this.timestampOffset !== 0) {
  (this.#sourceBuffer as any).timestampOffset = this.timestampOffset;
}
```

For real SourceBuffers this shifts the browser's decode-time-to-presentation
mapping. For `TextSourceBuffer` it is stored and applied during cue
construction.

Also clear it in `sourceBuffer` setter (alongside the other resets) so a
re-assigned buffer starts clean.

### 3. ManagedSourceBuffer — expose timestampOffset

```typescript
get timestampOffset(): number { return this.sourceBuffer.timestampOffset; }
set timestampOffset(v: number) { this.sourceBuffer.timestampOffset = v; }
```

### 4. ISourceBuffer interface — add timestampOffset

```typescript
timestampOffset: number;
```

### 5. TextSourceBuffer — apply timestampOffset in cue construction

`timestampOffset` is a plain `number` property (default `0`). Applied in
`append()` when converting fMP4 decode times to cue presentation times:

```typescript
const presentationTime = sample.pts + this.timestampOffset;
// sample.pts is already divided by timescale: DT / timescale
```

## STPP support scope

V1 TTML/IMSC support (`stpp`): extract `<p>` elements via `DOMParser`, reading
`begin`/`end` timing attributes (SMPTE clock-time format
`HH:MM:SS[.fraction]`). Strip all TTML region/styling markup — cue text is
the concatenated text content of each `<p>`. This covers the common case of
broadcast subtitle TTML and most OTT TTML distributions. Image-based TTML
(IMSC profile with `<image>` samples) is explicitly out of scope for V1 and
logged as a warning.

## Subtitle visibility default

The `TextTrack` starts in `'hidden'` mode. The "None" ADS is active by default,
so `TextSourceBuffer.hide()` is called on first activation, confirming `hidden`.
The user selects a language track from the subtitle menu to switch to `'showing'`.
