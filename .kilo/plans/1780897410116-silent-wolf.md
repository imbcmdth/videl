# Plan: Unified Wall-Clock Time Coordinate System

**Goal:** Replace the current fragmented timing model — where live representations compute their own `timestampOffset`, apply it to segment `startTime` values as a currentTime-space hack, and scatter time-space conversion throughout the tree — with a single, coherent model where all times in the component tree are **wall-clock epoch seconds**, and only `videl-player` and `ManagedSourceBuffer` know how to map those to the browser's `video.currentTime` space.

---

## Background and Problem Summary

The current model has three active bugs we've been fighting:

1. `#updateLiveSegments` used to store segment `startTime` as `periodStart + i * segDurSec` (a large epoch-relative number) while `currentTime` is in `[0, TSBD]` → we patched it by applying `tsOffset` inline.
2. The `tsOffset` was only available after `#computeLiveTimestampOffset` ran at activation — creating an ordering dependency between activation and segment population.
3. `#updateLiveSeekableRange` in `videl-player` re-derives the live edge from `availabilityStartTime + timestampOffset` — it already had this information but had to reach into a child representation to get it.

The root cause: responsibility for the `currentTime ↔ wall-clock` mapping is spread across `videl-representation`, `videl-player`, and `#updateLiveSegments`. No single owner.

---

## The Proposed Model

### Core invariant

```
wallAnchor  =  wall-clock epoch seconds corresponding to video.currentTime = 0
wallClock   =  video.currentTime + wallAnchor
```

- **VOD:** `wallAnchor = 0`. So `wallClock = currentTime`. Epoch seconds and presentation seconds are identical (the existing behaviour, unchanged).
- **live (TSBD = 0):** `wallAnchor = activationNow - 0 = activationNow`. The live edge maps to `currentTime ≈ 0`.
- **live-dvr:** `wallAnchor = activationNow - TSBD`. DVR window start = `currentTime = 0`, live edge = `currentTime = TSBD`.

Formula matches ADR-0005 exactly — only the *owner* moves.

### Who owns what

| Concern | Owner (new) | Owner (old) |
|---|---|---|
| Compute `wallAnchor` | `videl-player` | scattered (`videl-representation`) |
| Stamp `wallAnchor` on `ManagedSourceBuffer` | `videl-player` | n/a |
| Translate wall-clock → `SourceBuffer.timestampOffset` | `ManagedSourceBuffer` | `videl-representation` |
| Segment `startTime` space | wall-clock epoch seconds | currentTime (after tsOffset hack) |
| `PlayerState` time | `wallClock` (epoch) | `currentTime` (video-relative) |
| `buffered` TimeRanges exposed to pump | epoch-shifted (via `OffsetTimeRanges`) | currentTime-space |

## Detailed Changes

### 1. `videl-player` — compute and own `wallAnchor`

At MSE `sourceopen` (where `MediaSource.duration` is currently set), compute `wallAnchor`:

```ts
// Live: anchor to DVR-window start at this moment
const tsbd = effectiveTsbd;                       // from active representation
const availStart = ...;                            // from active representation
wallAnchor = tsbd > 0
  ? Date.now() / 1000 - tsbd          // live-dvr: window start = currentTime 0
  : Date.now() / 1000;                // live (no window): live edge = currentTime 0

// VOD
wallAnchor = 0;
```

Store `wallAnchor` on the player instance. Stamp it as a `wall-anchor` DOM attribute so the tree can read it if needed.

Pass it to each `ManagedSourceBuffer` after construction:
```ts
msb.wallAnchor = wallAnchor;
```

### 2. `ISourceBuffer` / `ManagedSourceBuffer` / `TextSourceBuffer` — complete wall-clock interface

#### Full `ISourceBuffer` surface (what the interface will expose)

The interface currently covers `append`, `remove`, `abort`, `changeType`, `updating`, `buffered`, `timestampOffset`, and optional `show`/`hide`. Three members from the real `SourceBuffer` API are missing and must be added, and two existing members change semantics:

| Member | Status | Notes |
|---|---|---|
| `wallAnchor: number` | **New** | Set by `videl-player` after construction. Wall-clock epoch second at `currentTime = 0`. |
| `timestampOffset` | **Semantics change** | Callers now pass a wall-clock epoch offset. Implementations subtract `wallAnchor` before applying to the real buffer. |
| `remove(start, end)` | **Semantics change** | Arguments are wall-clock epoch seconds. Implementations subtract `wallAnchor` before the real call. |
| `buffered` | **Semantics change** | Returns epoch-shifted ranges (`+wallAnchor`). |
| `appendWindowStart` | **New** | Wall-clock epoch seconds. Queued; implementations subtract `wallAnchor` for the real buffer. |
| `appendWindowEnd` | **New** | Wall-clock epoch seconds. Queued; same translation. |
| `mode` | **New** | `'segments' \| 'sequence'`. Synchronous pass-through, no translation. |
| `append`, `abort`, `changeType`, `updating`, `show?`, `hide?` | Unchanged | |

#### `ManagedSourceBuffer` implementation details

**New field:**
```ts
wallAnchor = 0;  // set by videl-player after addSourceBuffer
```

**`timestampOffset` setter** — translates to player-time before queuing:
```ts
set timestampOffset(wallOffset: number) {
  this.queue.push({ kind: 'timestampOffset', value: wallOffset - this.wallAnchor, ... });
  this.processQueue();
}
```
VOD (`wallAnchor = 0`): `value = wallOffset` — identical to today.  
Live-dvr: `value = availStart - (activationNow - TSBD) = availStart + TSBD - activationNow` — matches ADR-0005.

**`remove(start, end)` — translates arguments:**
```ts
async remove(start: number, end: number): Promise<void> {
  const a = start - this.wallAnchor;
  const b = end === Infinity ? Infinity : end - this.wallAnchor;
  this.queue.push({ kind: 'remove', start: a, end: b, ... });
}
```

**`appendWindowStart` / `appendWindowEnd` setters** — queued, translated:
```ts
set appendWindowStart(wallValue: number) {
  this.queue.push({ kind: 'appendWindowStart', value: wallValue - this.wallAnchor, ... });
}
set appendWindowEnd(wallValue: number) {
  this.queue.push({ kind: 'appendWindowEnd', value: wallValue - this.wallAnchor, ... });
}
```
Getters return the current real `SourceBuffer` value (not re-translated — clients generally set and forget).

**`mode` getter/setter** — synchronous pass-through to `sourceBuffer.mode`. Added to `MsbQueueEntry` union for ordering safety (same pattern as `changeType` and `timestampOffset`).

**`buffered` getter** — epoch-shifted:
```ts
get buffered(): TimeRanges {
  return new OffsetTimeRanges(this.sourceBuffer.buffered, this.wallAnchor);
}
```

#### `TextSourceBuffer` implementation details

`VTTCue.startTime` / `VTTCue.endTime` must be in `currentTime` space (the browser matches them against `video.currentTime`). So `TextSourceBuffer` stores and applies times in currentTime space internally, exposes epoch space externally — symmetric with `ManagedSourceBuffer`.

**New field:**
```ts
wallAnchor = 0;
```

**`timestampOffset` — change from plain field to computed:**
```ts
#wallTimestampOffset = 0;
get timestampOffset() { return this.#wallTimestampOffset; }
set timestampOffset(wallOffset: number) {
  this.#wallTimestampOffset = wallOffset;
  // effective player-time offset = wallOffset − wallAnchor
  // (used below in cue time calculations)
}
```
All internal uses of `this.timestampOffset` in `#doAppend` change to `this.#wallTimestampOffset - this.wallAnchor`.

**`remove(start, end)` — translates before removing cues and cutting bufferedRanges:**
```ts
async remove(start: number, end: number): Promise<void> {
  const a = start - this.wallAnchor;
  const b = end === Infinity ? Infinity : end - this.wallAnchor;
  // queue translated (a, b)
}
```

**`appendWindowStart` / `appendWindowEnd`** — stored in currentTime space; filter cues during `#doAppend`:
```ts
#appendWindowStart = -Infinity; // player-time
#appendWindowEnd   =  Infinity;

set appendWindowStart(wallValue: number) {
  this.#appendWindowStart = wallValue - this.wallAnchor;
}
set appendWindowEnd(wallValue: number) {
  this.#appendWindowEnd = wallValue - this.wallAnchor;
}
```
Cues with `presentationTime < #appendWindowStart || presentationTime >= #appendWindowEnd` are silently discarded.

**`mode`** — no-op setter, getter returns `'segments'` (cue times are always absolute).

**`buffered` getter** — epoch-shift the internal `SyntheticTimeRanges`:
```ts
get buffered(): TimeRanges {
  return new OffsetTimeRanges(this.#bufferedRanges as unknown as TimeRanges, this.wallAnchor);
}
```

#### New helper: `OffsetTimeRanges`

```ts
// src/lib/ergo-mse/offset-time-ranges.ts
export class OffsetTimeRanges {
  constructor(private inner: TimeRanges, private offset: number) {}
  get length() { return this.inner.length; }
  start(i: number) { return this.inner.start(i) + this.offset; }
  end(i: number)   { return this.inner.end(i)   + this.offset; }
}
```

### 3. `PlayerState` — add `currentWallTime`; deprecate `currentTime`

```ts
interface PlayerState {
  /** wall-clock epoch seconds = video.currentTime + wallAnchor. Primary time field. */
  currentWallTime: number;
  wallAnchor:      number;
  /**
   * Raw video.currentTime. Kept for logging/debugging only.
   * @deprecated Use currentWallTime for all component logic.
   */
  currentTime: number;
  /** Epoch-shifted buffered ranges (video.buffered + wallAnchor). */
  buffered: TimeRanges;   // wrapped with OffsetTimeRanges — see decision A2
  // ... rest unchanged
}
```

In `videl-player.#pumpTick`:
```ts
const rawTime        = this.#video.currentTime;
const currentWallTime = rawTime + this.#wallAnchor;
const state: PlayerState = {
  currentWallTime,
  currentTime: rawTime,          // deprecated — logging only
  wallAnchor:  this.#wallAnchor,
  buffered:    new OffsetTimeRanges(this.#video.buffered, this.#wallAnchor),
  ...
};
```

All pump consumers (`videl-representation.videlUpdate`, `videl-period.videlUpdate`, etc.) use `state.currentWallTime`. `video.currentTime` (for seeks, position restore) is still read/written directly on `this.#video` inside the player. The public `VidelPlayer.get buffered()` continues to return raw `video.buffered` (external HTMLMediaElement proxy contract, unchanged).

### 4. `videl-representation` — simplify to pure epoch arithmetic

**Remove `#computeLiveTimestampOffset` entirely.** The live `timestampOffset` is now set by `videl-player` on the `ManagedSourceBuffer`, not by the representation.

**Remove `tsOffset` from `#updateLiveSegments`.** Segment `startTime` is just:
```ts
const sTime = availabilityStartTime + periodStart + i * segDurSec;
```
No `tsOffset` needed — it's pure DASH epoch arithmetic.

**Fix extend path** (no longer needs to subtract tsOffset):
```ts
const lastRelSec = lastSeg.startTime - availabilityStartTime - periodStart;
const lastSegNum = startNum + Math.round(lastRelSec / segDurSec);
```

**Fix `#updateLiveSegments` expiry / latestSegNum / fromSegNum** — all already use `availabilityStartTime + periodStart` (from the recent bug-fix commit), no change needed.

**`videlUpdate` pump**: replace all `currentTime` references with `state.wallClock`. All `isBuffered` / `bufferedAhead` / `bufferEndNear` calls work against epoch-shifted `sourceBuffered` ranges — no conversion needed.

**Seek detection**: delta threshold is in seconds, so it's coordinate-space-independent — no change.

**`#timelineDrift`**: works identically in epoch space. `actualEnd = bufferEndNear(effectiveStart, sbBuffered)` where both are epoch-space — consistent.

### 5. Parser — stamp wall-clock period starts for live

The MPD parser currently stamps `Period@start` as a duration offset (seconds from period 0), which equals the presentation-relative start. For live streams, the absolute wall-clock start would be `availabilityStartTime + periodStart`. The parser should stamp this on each period for live:

```ts
// In buildPeriod, for live (liveCtx present):
const wallStart = liveCtx
  ? liveCtx.availabilityStartTime + start
  : start;
el.setAttribute('start', String(wallStart));
```

For VOD, `wallStart = 0 + start = start` — unchanged.

This means `periodStart` read by representations is already in epoch seconds for live, and already 0-relative for VOD — no conversion needed anywhere downstream.

### 6. `#updateLiveSeekableRange` in `videl-player` — simplify

With `wallAnchor` known by the player:

```ts
const nowSec    = Date.now() / 1000;
const liveEdge  = nowSec - this.#wallAnchor;   // in currentTime space
const start     = liveEdge - tsbd;
ms.setLiveSeekableRange(Math.max(0, start), liveEdge);
```

No need to reach into a child representation for `availability-start-time` or `timestamp-offset` — the player has everything it needs. The computation is the same math, cleaner access.

### 7. Update ADR-0005

Add a section describing the unified wall-clock model, `wallAnchor` ownership, and the new `ManagedSourceBuffer.timestampOffset` semantics. Update the `timestampOffset` derivation table to show callers pass wall-clock offsets.

## Decisions (Agreed)

**A1 — `PlayerState.currentTime` naming:**  
Field is renamed to `currentWallTime`. The old `currentTime` field is kept alongside it, stamped with the raw `video.currentTime` value, and marked `@deprecated` — available for logging and debugging but no component logic should read it. All components are updated to use `currentWallTime`.

**A2 — `PlayerState.buffered`:**  
Keep but epoch-shift via `OffsetTimeRanges(video.buffered, wallAnchor)`. Note: the public `VidelPlayer.get buffered()` proxy (HTMLMediaElement surface) continues to return raw `video.buffered` in currentTime-space — that is an external API contract and must not change.

**A3 — `appendWindowStart` / `appendWindowEnd`:**  
Treated identically to `timestampOffset`. Callers provide wall-clock epoch seconds; `ManagedSourceBuffer` translates to actual SourceBuffer player-time by subtracting `wallAnchor`. Both properties are added to `ISourceBuffer` and implemented in `ManagedSourceBuffer` and `TextSourceBuffer`. Not currently used anywhere — this adds the correct abstraction for future use.

## What Does NOT Change

- `SourceBuffer.timestampOffset` in the browser is still set by `ManagedSourceBuffer` — just computed differently.
- VOD behaviour is identical — `wallAnchor = 0` makes all formulas degenerate to today's values.
- `setLiveSeekableRange` is still called from `videl-player` (already correct owner per ADR-0005).
- `#timelineDrift` mechanism is unchanged — it just operates in epoch seconds instead of currentTime seconds.
- The DASH segment number / availability time formulas are unchanged — they already use `availabilityStartTime + periodStart`.
