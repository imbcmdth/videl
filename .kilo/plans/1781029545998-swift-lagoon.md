# MPD Event Handling â€” `<videl-event-stream>` + `<videl-event>`

## Overview

DASH MPD events (`EventStream`/`Event` in the manifest XML) need a DOM-first
implementation that mirrors how media tracks work today:

```
<videl-period>
  <videl-adaptation-set content-type="video">â€¦</videl-adaptation-set>
  <videl-adaptation-set content-type="audio">â€¦</videl-adaptation-set>
  <videl-event-stream scheme-id-uri="â€¦" timescale="â€¦" value="â€¦">
    <videl-event presentation-time="â€¦" duration="â€¦" id="â€¦">payload</videl-event>
    <videl-event â€¦/>
  </videl-event-stream>
</videl-period>
```

`<videl-event-stream>` is the Representation analogue â€” it holds the EventStream
context and owns an `EventSourceBuffer` from ergo-mse.  `<videl-event>` is the
segment analogue â€” each one serialises itself as XML and calls
`sourceBuffer.append()` when activated.

The `EventSourceBuffer` (defined in ergo-mse) accepts serialised DASH XML:
- A `<EventStream â€¦>` wrapper sets the buffer's context (schemeIdUri, value,
  timescale, presentationTimeOffset) â€” exactly like an init segment.
- Bare `<Event â€¦>` elements are processed against the previously-set context â€”
  exactly like media segments.

emsg in-band events are handled by ergo-mse transparently once
`ManagedSourceBuffer.enableEventDetection(sink)` is wired (which this plan also
covers).  This plan is focused on MPD events.

---

## Relationship to ergo-mse plan

The ergo-mse plan (`mpeg-dash-events-support.md`) adds:
- `EventSourceBuffer` (new `ISourceBuffer` implementation)
- `EventSink`, `BoundedEventDedup`, `DashEventEmitter`
- `parseEmsgBoxes` for in-band emsg v1 detection
- `parseEventStreamXml` for the XML append path
- `classifyTextMimeAndCodecs` recognises `application/dash+xml; codecs="event-stream"`
- `ErgoMediaSource.addSourceBuffer` routes that MIME type to an `EventSourceBuffer`
- `ManagedSourceBuffer.enableEventDetection(sink)` for emsg scanning

This videl plan assumes those ergo-mse additions are landed and available via the
`ergo-mse` package import.

for (const es of children(period, 'EventStream')) {

### 1.2 `buildEventStream`

```typescript
function buildEventStream(
  es:  Element,
  ctx: { periodStart: number; base: string }
): HTMLElement {
  const el = document.createElement('videl-event-stream');

  // Stamp EventStream-level context attributes (raw MPD values; element
  // converts to numeric during activation like timestampOffset).
  el.setAttribute('scheme-id-uri',             es.getAttribute('schemeIdUri') ?? '');
  const timescale = Number(es.getAttribute('timescale') ?? '1') || 1;
  const pto       = Number(es.getAttribute('presentationTimeOffset') ?? '0');
  el.setAttribute('timescale',                 String(timescale));
  el.setAttribute('presentation-time-offset',  String(pto));
  if (es.hasAttribute('value')) {
    el.setAttribute('value', es.getAttribute('value')!);
  }
  el.setAttribute('period-start', String(ctx.periodStart));

  // Build one <videl-event> per Event child.
  for (const ev of children(es, 'Event')) {
    el.appendChild(buildEvent(ev));
  }

  return el;
}

function buildEvent(ev: Element): HTMLElement {
  const el = document.createElement('videl-event');
  if (ev.hasAttribute('presentationTime')) {
    el.setAttribute('presentation-time', ev.getAttribute('presentationTime')!);
  }
  if (ev.hasAttribute('duration')) {
    el.setAttribute('duration', ev.getAttribute('duration')!);
  }
  if (ev.hasAttribute('id')) {
    el.setAttribute('id', ev.getAttribute('id')!);
  }
  if (ev.hasAttribute('messageData')) {
    el.setAttribute('message-data', ev.getAttribute('messageData')!);
  }
  // Preserve inline payload as text content.
  el.textContent = ev.textContent?.trim() ?? '';
  return el;
}
```

**Attribute naming:** follows the project convention â€” no `videl-` prefix on
manifest-derived attributes (`scheme-id-uri`, `timescale`,
`presentation-time-offset`, `value`, `period-start`, `presentation-time`,
`duration`, `id`, `message-data`).

## Phase 2 â€” `<videl-event>` Element

`src/elements/videl-event.ts`

Analogous to `<videl-segment>` without the fetch lifecycle. Holds event data as
attributes and serialises itself to XML bytes when activated.

```typescript
export class VidelEvent extends LitElement {
  static properties = {
    presentationTime: { type: Number, attribute: 'presentation-time' },
    duration: { type: Number },
    eventId: { type: Number, attribute: 'id' },
    messageData: { type: String, attribute: 'message-data' },
  };

  presentationTime = 0;
  duration: number | null = null;
  eventId: number | null = null;
  messageData: string | null = null;

  sourceBuffer: ISourceBuffer | null = null;  // set by parent VidelEventStream

  // Observed: videl-state
  attributeChangedCallback(name, old, value) { â€¦ }

  #onActivate(): void {
    // Serialise this event as XML bytes and append to the EventSourceBuffer.
    const xml = this.#toXml();
    const bytes = new TextEncoder().encode(xml);
    this.sourceBuffer?.append(bytes).then(() => {
      this.setAttribute('videl-done', '');
      this.dispatchEvent(new CustomEvent('videl:done', {
        bubbles: true, composed: true,
        detail: { presentationTime: this.presentationTime }
      }));
    }).catch(err => {
      this.dispatchEvent(new CustomEvent('videl:segment:error', {
        bubbles: true, composed: true,
        detail: { error: err }
      }));
    });
  }

  #toXml(): string {
    const pt   = this.getAttribute('presentation-time') ?? '0';
    const dur  = this.hasAttribute('duration') ? ` duration="${this.getAttribute('duration')}"` : '';
    const id   = this.hasAttribute('id')       ? ` id="${this.getAttribute('id')}"`             : '';
    const md   = this.messageData              ? ` messageData="${this.messageData}"`            : '';
    const body = this.messageData ? '' : (this.textContent?.trim() ?? '');
    return `<Event presentationTime="${pt}"${dur}${id}${md}>${body}</Event>`;
  }
}
```

**Key behaviours:**
- `videl-state="active"` â†’ call `#onActivate()`, append XML, fire `videl:done`
- `videl-state` removed â†’ no-op (events aren't abortable; they're synchronous)
- `videl-done` attribute stamped after successful append (DOM-observable state)
- No `videl-state="next"` action needed (no prefetch â€” there is nothing to fetch)

## Phase 3 â€” `<videl-event-stream>` Element

`src/elements/videl-event-stream.ts`

Analogous to `<videl-representation>`, minus ABR and init-segment fetching.

### Attributes

| Attribute | Origin | Description |
|-----------|--------|-------------|
| `scheme-id-uri` | manifest | `EventStream@schemeIdUri` |
| `value` | manifest | `EventStream@value` (optional) |
| `timescale` | manifest | tick rate; default 1 |
| `presentation-time-offset` | manifest | PTO in timescale ticks; default 0 |
| `period-start` | manifest (computed) | Wall-clock epoch seconds of the period start |
| `videl-state` | internal state | `"next"` â†’ `"active"` â†’ absent |
| `videl-context-sent` | internal state | Present once the EventStream context append has been made |

### Lifecycle

```typescript
export class VidelEventStream extends LitElement {
  // Properties mirror attributes listed above.

  sourceBuffer: ISourceBuffer | null = null;

  attributeChangedCallback(name, old, value) {
    super.attributeChangedCallback(name, old, value);
    if (name !== 'videl-state') return;
    if (value === 'active' || value === 'next') {
      this.#ensureContextSent();
    } else if (value === null) {
      // Reset so a new activation re-sends context (e.g. after period switch).
      this.removeAttribute('videl-context-sent');
      // Cascade: clear videl-done + videl-state from child events.
      for (const ev of this.#childEvents) {
        ev.removeAttribute('videl-done');
        ev.removeAttribute('videl-state');
      }
    }
  }

  /**
   * Called by videl-period on each pump tick (when active).
   * Walk child events and append those that are not yet done.
   */
  videlUpdate(state: PlayerState): void {
    if (this.getAttribute('videl-state') !== 'active') return;
    this.#dispatchPendingEvents(state.currentWallTime, state.bufferAhead);
  }

  /** Send the EventStream context (init-segment equivalent). */
  #ensureContextSent(): void {
    if (this.hasAttribute('videl-context-sent') || !this.sourceBuffer) return;
    const xml   = this.#contextXml();
    const bytes = new TextEncoder().encode(xml);
    this.sourceBuffer.append(bytes).catch(err =>
      console.warn('[videl-event-stream] context append failed:', err)
    );
    this.setAttribute('videl-context-sent', '');
  }

  /**
   * Activate events that haven't been appended yet.
   *
   * Time-window policy: only append events whose resolved wall-clock
   * presentationTime falls within [currentWallTime - 1, currentWallTime + bufferAhead].
   *
   * The lower bound (-1 s) catches events that just became current.
   * The upper bound (bufferAhead) matches the media buffer-ahead window, so
   * events are delivered slightly ahead of the playhead exactly like segments.
   *
   * For live MPD refresh: new <videl-event> children added to the DOM after
   * activation are picked up naturally on the next videlUpdate tick, since the
   * walk always iterates the live child list.
   */
  #dispatchPendingEvents(currentWallTime: number, bufferAhead: number): void {
    if (!this.hasAttribute('videl-context-sent')) return;
    const timescale = Number(this.getAttribute('timescale') ?? '1') || 1;
    const pto       = Number(this.getAttribute('presentation-time-offset') ?? '0');
    const periodStart = Number(this.getAttribute('period-start') ?? '0');
    for (const ev of this.#childEvents) {
      if (ev.hasAttribute('videl-done') || ev.getAttribute('videl-state') === 'active') continue;
      // Resolve to wall-clock seconds (same formula as ergo-mse EventSourceBuffer).
      const rawPt  = Number(ev.getAttribute('presentation-time') ?? '0');
      const wallPt = (rawPt - pto) / timescale + periodStart;
      if (wallPt < currentWallTime - 1 || wallPt > currentWallTime + bufferAhead) continue;
      ev.sourceBuffer = this.sourceBuffer;
      ev.setAttribute('videl-state', 'active');
    }
  }

  get #childEvents(): VidelEvent[] {
    return Array.from(this.children).filter(
      el => el.tagName.toLowerCase() === 'videl-event'
    ) as VidelEvent[];
  }

  /** Produce the EventStream wrapper XML (no Event children â€” context only). */
  #contextXml(): string {
    const scheme = this.getAttribute('scheme-id-uri') ?? '';
    const ts     = this.getAttribute('timescale') ?? '1';
    const pto    = this.getAttribute('presentation-time-offset') ?? '0';
    const value  = this.hasAttribute('value') ? ` value="${this.getAttribute('value')}"` : '';
    return `<EventStream schemeIdUri="${scheme}" timescale="${ts}" presentationTimeOffset="${pto}"${value}/>`;
  }
}
```

**Timing note:** The ergo-mse `EventSourceBuffer` converts raw `presentationTime`
ticks to wall-clock seconds using the formula:
```
wallClock = (presentationTime - presentationTimeOffset) / timescale + effectiveOffset
```
where `effectiveOffset = wallTimestampOffset âˆ’ wallAnchor` is the period's wall
start translated into the MSE coordinate system.  The event stream element must
set `sourceBuffer.timestampOffset` to `periodStart` so this calculation resolves
correctly â€” same as `videl-representation` sets it after the init segment.

Concretely, after `#ensureContextSent()`:
```typescript
this.sourceBuffer.timestampOffset = Number(this.getAttribute('period-start') ?? '0');
```

## Phase 4 â€” `videl-period` Integration

### 4.1 Activating event streams

`videl-period` currently iterates `<videl-adaptation-set>` children only. It needs
to also activate / deactivate `<videl-event-stream>` children. Since event
streams have no selection logic (all are always active), they're handled as a
separate pass â€” not through `PickNMixin`.

```typescript
// New helper in VidelPeriod:
get #childEventStreams(): VidelEventStream[] {
  return Array.from(this.children).filter(
    el => el.tagName.toLowerCase() === 'videl-event-stream'
  ) as VidelEventStream[];
}
```

In `#activateAll()`:
```typescript
// After activating adaptation sets:
for (const es of this.#childEventStreams) {
  es.setAttribute('videl-state', 'active');
}
```

In `#preloadAll()`:
```typescript
// Preload = send context early so events are ready.
for (const es of this.#childEventStreams) {
  es.setAttribute('videl-state', 'next');
}
```

In `attributeChangedCallback` when `videl-state` is removed:
```typescript
// After clearing adaptation sets (PickNMixin handles those):
for (const es of this.#childEventStreams) {
  es.removeAttribute('videl-state');
}
```

### 4.2 Pump forwarding

In `videlUpdate()`, after the existing adaptation-set fan-out:
```typescript
for (const es of this.#childEventStreams) {
  if (es.getAttribute('videl-state') === 'active') {
    es.videlUpdate(state);
  }
}
```

Event streams are excluded from period-completion logic (same exclusion that text
tracks have â€” only video + audio gate `videl:done`).

## Phase 5 â€” `videl-player` Integration

### 5.1 Create the EventSourceBuffer

At MSE setup time (`#setupMse`), after iterating `<videl-adaptation-set>` children
to create media/text source buffers, create a single shared `EventSourceBuffer`
if any `<videl-event-stream>` elements exist in the presentation tree.

```typescript
const hasEventStreams = !!presEl.querySelector('videl-event-stream');
let eventSourceBuffer: ISourceBuffer | null = null;

if (hasEventStreams) {
  try {
    eventSourceBuffer = mse.addSourceBuffer(
      'application/dash+xml; codecs="event-stream"'
    );
    eventSourceBuffer.wallAnchor = this.#wallAnchor;
    this.#sourceBuffers.set('event-stream', eventSourceBuffer);
    trace(this, 'mse', 'add-event-source-buffer', {});
  } catch (e) {
    console.warn('[videl-player] EventSourceBuffer creation failed:', e);
  }
}

// Assign to all event-stream elements.
if (eventSourceBuffer) {
  for (const es of presEl.querySelectorAll('videl-event-stream') as NodeListOf<VidelEventStream>) {
    es.sourceBuffer = eventSourceBuffer;
  }
}
```

The `EventSourceBuffer` is a fake/virtual buffer (like `TextSourceBuffer`), so it
doesn't affect the `endOfStream` gate. The existing `isValidContentType` guard in
the adaptation-set loop already skips `content-type="event-stream"` if we were to
stamp that on event streams; since event streams are not adaptation sets this is
moot, but it's worth noting for clarity.

### 5.2 emsg detection wiring

No explicit wiring needed in videl. `ErgoMediaSource.addSourceBuffer()` calls
`#ensureEventInfrastructure()` internally for both the event-stream MIME type
and real MSE (video/audio) buffers, and wires `ManagedSourceBuffer.enableEventDetection(sink)`
itself. Creating the `EventSourceBuffer` via `mse.addSourceBuffer('application/dash+xml; codecs="event-stream"')`
is sufficient â€” emsg scanning on all video/audio `ManagedSourceBuffer` instances
is activated automatically from that point.

### 5.3 MSE teardown

In `#teardownMse()`, clear event source buffer cues alongside text source buffer
cues (or rely on the EventSourceBuffer's own `remove(0, Infinity)` if it
implements that). Confirm behaviour with ergo-mse's `EventSourceBuffer.remove()`
contract.

## Open Questions / Design Decisions

The following need resolution before or during implementation:

1. **`sourceBuffer.timestampOffset` for the event stream:**
   The ergo-mse plan describes `effectiveOffset = wallTimestampOffset âˆ’ wallAnchor`.
   Confirm that setting `eventSourceBuffer.timestampOffset = periodStart` on
   `VidelEventStream` is the correct way to supply this value, or whether
   `wallAnchor` alone is sufficient for the `EventSourceBuffer` to resolve times.

2. **Live MPD refresh â€” element readiness (in scope) vs parser wiring (follow-on):**
   `VidelEventStream.#dispatchPendingEvents` always iterates the live child list,
   so any `<videl-event>` appended to the DOM after activation is picked up on the
   next `videlUpdate` tick â€” no special handling needed in the element.
   The MPD parser refresh path (where `videl-presentation` re-fetches the MPD and
   needs to diff/merge new `<videl-event>` children into existing
   `<videl-event-stream>` elements) is **not** part of this plan; it is a follow-on
   that depends on the MPD refresh feature being implemented first.

3. **`<videl-event>` `videl-state="next"` (prefetch):**
   Currently no action is taken on `next`. Should a future optimisation allow
   pre-serialising the XML? Probably unnecessary â€” the serialisation is trivial.
   Keeping `next` as a no-op is fine.

4. **EventSourceBuffer shared across periods â€” dedup handles overlap:**
   The `EventSourceBuffer` is shared across periods (like `TextSourceBuffer`).
   The ergo-mse `BoundedEventDedup` in the shared `EventSink` prevents double-fires
   on period transitions without any explicit `remove()` call. Old cues remain in
   the TextTrack but VTTCue timing means they won't re-fire once the playhead
   has passed them. No teardown action needed at period deactivation.
