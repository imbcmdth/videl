# Plan: DRM Support with EME

## Summary

Add Encrypted Media Extensions (EME) support to the videl player, enabling playback of Widevine, PlayReady, and ClearKey protected DASH content. This covers MPD `ContentProtection` parsing, EME session management, and license acquisition.

> **LEMMA context**: LEMMA.md §V1 Scope declares "no DRM" as a V1 constraint; this is a planned post-V1 feature. All conventions from LEMMA apply: DOM-first data model, attribute naming rules, ES private fields, `videl-` prefix for internal state attributes only.

## Research Summary

### Codebase Architecture

```
videl-player          → owns <video>, MSE, pump loop; orchestrates everything
  videl-presentation  → owns MPD parse, period lifecycle, UI overlay
    videl-period      → owns one period's adaptation sets
      videl-adaptation-set → ABR, source buffer distribution
        videl-representation → segment fetching
mpd-parser.ts         → converts MPD XML → element tree
```

### DASH-IF Real-World DRM Patterns (from sources.json)

- Multiple DRM systems per stream (Widevine + PlayReady offered simultaneously; browser picks one)
- Key system IDs: `com.widevine.alpha`, `com.microsoft.playready`, `org.w3.clearkey`
- Per-key-system: `serverURL`, `httpRequestHeaders` map, optional `httpTimeout`
- Encryption schemes: `cenc` (AES-CTR) and `cbcs` (AES-CBC / pattern encryption)
- Single-key (all ADS share one key) and multi-key (separate key per ADS, e.g. audio vs video)
- Key rotation (new keys every N seconds, triggered by `encrypted` events mid-stream)
- Multi-period with DRM — ContentProtection may differ across periods

### MPEG-DASH Spec Corner Cases (ISO/IEC 23009-1:2022 §5.8.4.1)

- `ContentProtection` can appear at MPD, Period, and AdaptationSet level
- Fifth-edition additions: `@ref`/`@refId` allow MPD-level CP elements to be referenced from ADS level (avoid repeating large PSSH blobs on every ADS)
- `@robustness` attribute per key system (new in ed.5)
- `cenc:default_KID` attribute carries the hex key ID for pre-filtering before license fetch
- PSSH box is typically base64-encoded inside `<cenc:pssh>` child element
- `urn:mpeg:dash:mp4protection:2011` scheme with `@value="cenc"` or `@value="cbcs"` signals the encryption mode at the container level (separate from the DRM system descriptor)
- `urn:uuid:<system-guid>` identifies specific DRM systems (Widevine: `edef8ba9-79d6-4ace-a3c8-27dcd51d21ed`, PlayReady: `9a04f079-9840-4286-ab92-e65be0885f95`, ClearKey: `e2719d58-a985-b3c9-781a-b030af78d30e`)
- ContentProtection inheritance: MPD-level CP with `@refId` → ADS-level CP with `@ref` pointing to it

### Shaka / dash.js Patterns

- Both create `MediaKeys` once per presentation and set on the video element before playback
- Both handle the `encrypted` event per init segment / key period
- Shaka: `DrmEngine` singleton per presentation, manages all sessions
- dash.js: `ProtectionController` + `KeySystem` abstraction per system
- Both support custom license request filters (headers, body transformation)
- Both deduplicate license requests by `initDataType+initData` to avoid duplicate sessions for the same key

## Component Responsibility Allocation

| Component | DRM Responsibility |
|---|---|
| `mpd-parser.ts` | Parse `ContentProtection` elements; serialize key-system data to `videl-adaptation-set` attributes; dereference `@ref/refId` |
| `videl-adaptation-set.ts` | Store `protection` data attribute; expose parsed CP info to player |
| `videl-player.ts` | **EME orchestrator**: `requestMediaKeySystemAccess`, `MediaKeys` creation, `video.setMediaKeys`, `encrypted` event handling, session management, license fetch |
| `videl-presentation.ts` | Accept `drm-config` attribute/property (per-presentation license server overrides); no EME logic |
| `videl-period.ts` | No DRM changes |
| `videl-representation.ts` | No DRM changes (encrypted segments are opaque to MSE once keys are set) |

> **Key decision**: EME lives in `videl-player` because it requires direct access to `this.#video` (for `setMediaKeys` and the `encrypted` event). The player already owns all video-element interactions.

## Implementation Plan

### Step 1 — MPD Parser: Extract ContentProtection

**File**: `src/parser/mpd-parser.ts`

1. In `buildAdaptationSet()`, collect all `ContentProtection` child elements of the `<AdaptationSet>` XML node.
2. Implement `resolveContentProtection(cp, mpdRoot)`:
   - If the element has `@ref`, look up the MPD-level `ContentProtection` element with matching `@refId` and merge attributes/children (ed.5 §5.2.3.5).
   - Extract `@schemeIdUri` (key system URN or `urn:mpeg:dash:mp4protection:2011`).
   - Extract `@value` (e.g. `"cenc"`, `"cbcs"`) for the protection scheme.
   - Extract `cenc:default_KID` if present (hex key ID, normalize to lowercase without dashes).
   - Extract `<cenc:pssh>` child text as base64 PSSH blob.
   - Extract `@robustness` if present.
3. Serialize as a JSON attribute on the `videl-adaptation-set` element:
   ```
   protection='[{"schemeIdUri":"urn:uuid:...","value":"cenc","keyId":"abc...","pssh":"base64...","robustness":""}]'
   ```
   Per the **LEMMA attribute naming convention**: this is DASH-manifest data stamped by the parser → **no `videl-` prefix** (same as `start`, `duration`, `bandwidth`, etc.).

**Corner cases**:
- Multiple `ContentProtection` elements on one ADS (one per supported DRM system + one for the scheme type)
- `@ref` pointing to MPD-level element (must pass `mpdRoot` down through `buildAdaptationSet`)
- Missing `<cenc:pssh>`: some streams signal protection in the init segment only (PSSH in `moov/pssh` box)

### Step 3 — DRM Configuration API

#### 3a. Type definitions (`src/index.ts`, exported)

Callbacks cannot go in attributes (strings only), so all DRM config lives exclusively as JS properties. Per LEMMA DOM-first: `protection` attributes capture what the MPD declares; `DrmConfig` captures what the consumer configures — these are different things at different layers.

```ts
interface DrmSystemConfig {
  serverUrl?: string;
  httpRequestHeaders?: Record<string, string>;
  httpTimeout?: number;

  // ── FairPlay: server certificate ─────────────────────────────────────────
  // Required for FairPlay. Must be installed via setServerCertificate() before
  // any EME session can be created. Provide exactly one of:
  certificateUrl?: string;   // player fetches automatically
  certificate?: Uint8Array;  // pre-fetched, skips the fetch

  // ── FairPlay: initData transform ─────────────────────────────────────────
  // FairPlay initData format is provider-defined (skd:// URL, UUID, binary
  // blob, custom URI, etc.). The raw bytes must be wrapped in Apple's binary
  // envelope before calling session.generateRequest().
  //
  // DEFAULT: treats initData as a UTF-8 skd:// URL, uses everything after
  //   "skd://" as contentId.
  // OVERRIDE: provide this when the provider uses a non-skd:// format or a
  //   custom contentId derivation.
  initDataTransform?: (
    initData: Uint8Array,
    initDataType: string,
    cert: Uint8Array | null
  ) => Uint8Array | Promise<Uint8Array>;

  // ── License response parser ───────────────────────────────────────────────
  // The HTTP response format is completely provider-defined. Common variants:
  //   • Raw binary bytes (Widevine, PlayReady, FairPlay raw CKC)
  //   • Base64-encoded bytes
  //   • JSON wrapper: {"license":"<b64>"} or {"ckc":"<b64>"}
  //   • XML wrapper (legacy PlayReady)
  //
  // DEFAULT: returns raw response bytes (correct for binary-only responses).
  // OVERRIDE: provide this for any provider that wraps or encodes the response.
  parseLicenseResponse?: (
    responseBody: ArrayBuffer
  ) => ArrayBuffer | Promise<ArrayBuffer>;

  // ── License request filter ────────────────────────────────────────────────
  // Mutate the outgoing request before it is sent: change URL, add headers,
  // wrap the SPC/challenge body, derive the license URL from the skd:// content
  // ID at runtime, etc.
  requestFilter?: (request: {
    url: string;
    headers: Record<string, string>;
    body: ArrayBuffer;
  }) => void | Promise<void>;
}

// Keyed by EME key system string, e.g. "com.widevine.alpha"
type DrmConfig = Record<string, DrmSystemConfig>;
```

#### 3b. Where consumers set config — two entry points, one lookup order

The player resolves DRM config by checking two places in order:

```
1. videl-presentation.drmConfig   (per-presentation, set in JS)
2. videl-player.drmConfig         (player-wide default, set in JS)
```

Both are plain JS properties on their respective elements. Neither is reflected as an attribute because callbacks cannot be serialized.

**Entry point 1: `videl-player.drmConfig` (JS property)**

Added to `VidelPlayer`. This is the **primary API** for `src`-attribute mode and for playlists where all presentations share the same DRM config. Set it before (or shortly after) setting `src` — DRM setup is async and runs after the init segment is appended:

```js
const player = document.querySelector('videl-player');
player.drmConfig = {
  'com.widevine.alpha': {
    serverUrl: 'https://license.example.com/widevine',
    parseLicenseResponse: async (body) => {
      const json = JSON.parse(new TextDecoder().decode(body));
      return Uint8Array.from(atob(json.license), c => c.charCodeAt(0)).buffer;
    }
  }
};
player.src = 'stream.mpd';
```

In `src`-attribute mode the player creates the `<videl-presentation>` internally — the consumer never has a reference to it. The player-level property is the only hook available.

**Entry point 2: `videl-presentation.drmConfig` (JS property)**

Added to `VidelPresentation`. Used in declarative playlist mode when individual presentations need different DRM configs (different license servers, different parsers per content). The consumer already holds references to the presentation elements and sets them in their JS setup:

```js
const pres1 = document.querySelector('#ep1');
pres1.drmConfig = {
  'com.widevine.alpha': { serverUrl: 'https://license-a.example.com/wv' }
};

const pres2 = document.querySelector('#ep2');
pres2.drmConfig = {
  'com.widevine.alpha': {
    serverUrl: 'https://license-b.example.com/wv',
    parseLicenseResponse: customParser
  }
};
```

**Timing guarantee**: `videl:before-activate` (defined in
`.kilo/plans/before-activate-event-and-play-semantics.md`) replaces the
"set `drmConfig` before `src`" timing requirement entirely. Consumers set
`drmConfig` inside the event handler and the player waits for them before
proceeding with activation:

```js
player.addEventListener('videl:before-activate', (event) => {
  if (event.detail.element.tagName === 'VIDEL-PRESENTATION') {
    event.waitUntil(
      fetchAuthToken().then(token => {
        event.detail.element.drmConfig = {
          'com.widevine.alpha': {
            serverUrl: 'https://license.example.com/wv',
            httpRequestHeaders: { Authorization: `Bearer ${token}` }
          }
        };
      })
    );
  }
});
```

> **See the before-activate plan** for `VidelBeforeActivateEvent`, `waitUntil`,
> `videl-state` on `<videl-player>`, and the `play()`/`pause()` attribute-driven
> rewrite. Those details are not repeated here.

**There is no `drm-config` attribute on `<videl-player>`.**

The previously planned JSON attribute is removed. It only carried serializable
config, which creates a confusing split between "attribute config" (no callbacks)
and "property config" (callbacks). One API surface is simpler:

```js
player.drmConfig = { 'com.widevine.alpha': { serverUrl: 'https://...' } };
```

#### 4b. `encrypted` listener lifetime

**Critical**: The `encrypted` event fires from `this.#video` at any time — not just once at stream start. Sources include:

- PSSH boxes in the **init segment** (`moov/pssh`): fires during the first `appendBuffer` call, before or during `#setupDrm`
- **In-band PSSH** in `moof` boxes mid-stream: key rotation on live streams (e.g. every 60 seconds per the AWS DRM reference streams in the DASH-IF samples)
- **Period transitions** with new ContentProtection: the new init segment is appended before the player has completed DRM re-setup for the new period
- **ABR switches** to a representation whose init segment carries different PSSH data

**Resolution**: Register the `encrypted` listener once in `connectedCallback` (and remove in `disconnectedCallback`), **not** inside `#setupMse`/`#teardownMse`. The handler must tolerate being called before `this.#mediaKeys` is ready:

```ts
// connectedCallback additions:
this.#video.addEventListener('encrypted', this.#onEncrypted);

// disconnectedCallback additions:
this.#video.removeEventListener('encrypted', this.#onEncrypted);
```

Add a pending-event queue for events that arrive before `MediaKeys` is available:

```ts
#pendingEncryptedEvents: MediaEncryptedEvent[] = [];
```

In `#onEncrypted`: if `this.#mediaKeys` is null, push to `#pendingEncryptedEvents` and return. After `setMediaKeys()` resolves in `#setupDrm`, drain the queue:

```ts
for (const evt of this.#pendingEncryptedEvents) {
  await this.#processEncryptedEvent(evt);
}
this.#pendingEncryptedEvents = [];
```

In `#teardownMse`, clear the pending queue (stale events from the previous presentation must not bleed into the next one):
```ts
this.#pendingEncryptedEvents = [];
```

#### 4c. `#setupMse()` modification

After creating source buffers, before setting `videl-state="active"`:

1. Collect all `protectionData` from all `videl-adaptation-set` elements in the presentation.
2. If any ADS has protection data:
   - Call `await this.#setupDrm(presEl, allProtectionData, signal)`.
   - After `setMediaKeys()` completes, drain `#pendingEncryptedEvents`.

#### 4d. `#setupDrm(presEl, protectionEntries, signal)`

```ts
async #setupDrm(presEl, entries, signal): Promise<void>
```

1. Read `drmConfig` from `presEl.drmConfig` (presentation-level) or `this.#drmConfig` (player-level).
2. Filter entries to those with `urn:uuid:` scheme (actual DRM systems, not the `urn:mpeg:dash:mp4protection:2011` scheme type entry).
3. Build `MediaKeySystemConfiguration[]` for each candidate key system.

   **For Widevine / PlayReady / ClearKey** (`initDataType: 'cenc'`):
   ```ts
   [{
     initDataTypes: ['cenc'],
     videoCapabilities: [{ contentType: 'video/mp4; codecs="..."', encryptionScheme }],
     audioCapabilities: [{ contentType: 'audio/mp4; codecs="..."', encryptionScheme }],
     persistentState: 'not-allowed',
     distinctiveIdentifier: 'not-allowed',
     sessionTypes: ['temporary']
   }]
   ```

   **For FairPlay** (`com.apple.fps`, `initDataType: 'skd'`):
   ```ts
   [{
     initDataTypes: ['skd'],  // ← different from cenc
     videoCapabilities: [{ contentType: 'video/mp4; codecs="avc1.42E01E"' }],
     audioCapabilities: [{ contentType: 'audio/mp4; codecs="mp4a.40.2"' }],
     persistentState: 'not-allowed',
     distinctiveIdentifier: 'not-allowed',
     sessionTypes: ['temporary']
   }]
   ```

4. Try key systems in preference order (Widevine first on Chrome, PlayReady on Edge, FairPlay on Safari):
   ```ts
   for (const keySystem of candidateKeySystems) {
     try {
       const access = await navigator.requestMediaKeySystemAccess(keySystem, configs);
       this.#mediaKeys = await access.createMediaKeys();
       await this.#video.setMediaKeys(this.#mediaKeys);
       this.#activeKeySystem = keySystem;
       break;
     } catch { continue; }
   }
   ```

5. **FairPlay only: fetch and install the server certificate.**  
   This must happen after `setMediaKeys()` and before any `generateRequest()` calls. Without the certificate, Safari will reject `generateRequest()` with a `NotSupportedError`.

   ```ts
   if (this.#isFairPlay()) {
     const fpConfig = this.#activeDrmSystemConfig();
     let cert = fpConfig?.certificate ?? null;
     if (!cert && fpConfig?.certificateUrl) {
       const resp = await fetch(fpConfig.certificateUrl, { signal });
       cert = new Uint8Array(await resp.arrayBuffer());
     }
     if (!cert) throw new Error('FairPlay requires a server certificate');
     await this.#mediaKeys!.setServerCertificate(cert);
     this.#fairPlayCert = cert;  // store for use in #processEncryptedEvent
   }
   ```

6. If no key system succeeded, dispatch `videl:drm:error` event and return (graceful degradation).

#### 4e. `encrypted` event handler

For Widevine/PlayReady, `initData` is a PSSH box and can be passed directly to `generateRequest`. For FairPlay, the `initData` is the raw `skd://` URL and **must be transformed** into the FairPlay envelope before calling `generateRequest`.

```ts
#onEncrypted = async (event: MediaEncryptedEvent): Promise<void> => {
  if (!this.#mediaKeys) {
    this.#pendingEncryptedEvents.push(event);
    return;
  }

  const { initDataType, initData } = event;

  // Deduplicate: hash initData to avoid duplicate sessions for same key
  const key = `${initDataType}:${hashInitData(initData)}`;
  if (this.#drmSessions.has(key)) return;

  // For FairPlay: transform initData into the required binary envelope before
  // calling generateRequest(). This is mandatory — Safari rejects the raw skd://
  // URL bytes as initData without the transformation.
  let processedInitData: BufferSource = initData;
  if (this.#isFairPlay() && initDataType === 'skd') {
    const config = this.#activeDrmSystemConfig();
    if (config?.initDataTransform) {
      processedInitData = await config.initDataTransform(
        new Uint8Array(initData),
        initDataType,
        this.#fairPlayCert
      );
    } else {
      // Default FairPlay transform: extract contentId from the skd:// URL
      processedInitData = this.#defaultFairPlayTransform(
        new Uint8Array(initData),
        this.#fairPlayCert
      );
    }
  }

  const session = this.#mediaKeys!.createSession('temporary');
  this.#drmSessions.set(key, session);

  session.addEventListener('message', this.#onKeyMessage);
  await session.generateRequest(initDataType, processedInitData);
};

/**
 * Default FairPlay initData envelope:
 *   [4 bytes LE] initDataSize
 *   [initDataSize bytes] initData (skd:// URL as UTF-8)
 *   [4 bytes LE] contentIdSize
 *   [contentIdSize bytes] contentId (the portion after skd://)
 *   [4 bytes LE] certSize
 *   [certSize bytes] cert
 */
#defaultFairPlayTransform(initData: Uint8Array, cert: Uint8Array | null): Uint8Array {
  const skdUrl = new TextDecoder().decode(initData);
  const contentId = skdUrl.startsWith('skd://') ? skdUrl.slice(6) : skdUrl;
  const contentIdBytes = new TextEncoder().encode(contentId);
  const certBytes = cert ?? new Uint8Array(0);

  const total = 4 + initData.length + 4 + contentIdBytes.length + 4 + certBytes.length;
  const result = new Uint8Array(total);
  const view   = new DataView(result.buffer);
  let offset   = 0;

  view.setUint32(offset, initData.length, true);  offset += 4;
  result.set(initData, offset);                   offset += initData.length;
  view.setUint32(offset, contentIdBytes.length, true); offset += 4;
  result.set(contentIdBytes, offset);             offset += contentIdBytes.length;
  view.setUint32(offset, certBytes.length, true); offset += 4;
  result.set(certBytes, offset);

  return result;
}
```

#### 4f. License acquisition (`message` event)

For FairPlay, `event.message` is the SPC (Server Playback Context). The license server returns a CKC (Content Key Context) in an entirely provider-defined format. Use `parseLicenseResponse` to decode the raw HTTP response body into the binary bytes for `session.update()`. The default is pass-through (raw bytes), which is correct for providers returning an unwrapped binary CKC.

For FairPlay the license server URL may be derived from the `skd://` URL rather than a fixed `serverUrl`. The default behavior is to use the configured `serverUrl`; implement a `requestFilter` to override the URL dynamically from the `skd://` content ID if needed.

```ts
#onKeyMessage = async (event: MediaKeyMessageEvent): Promise<void> => {
  const session = event.target as MediaKeySession;
  const config = this.#activeDrmSystemConfig();

  if (!config?.serverUrl) {
    this.dispatchEvent(new CustomEvent('videl:drm:error', {
      bubbles: true, composed: true,
      detail: { error: new Error('No license server URL configured') }
    }));
    return;
  }

  let body: ArrayBuffer = event.message;
  const headers: Record<string, string> = {
    'Content-Type': 'application/octet-stream',  // required for FairPlay; fine for others
    ...config.httpRequestHeaders
  };

  const reqContext = { url: config.serverUrl, headers, body };
  if (config.requestFilter) {
    await config.requestFilter(reqContext);
    body = reqContext.body;
  }

  const resp = await fetch(reqContext.url, {
    method: 'POST',
    headers: reqContext.headers,
    body,
    signal: this.#loadAbort?.signal
  });

  if (!resp.ok) {
    throw new Error(`License request failed: HTTP ${resp.status}`);
  }

  const rawBody = await resp.arrayBuffer();

  // Parse the license response into raw key bytes for session.update().
  // DEFAULT: return raw bytes unchanged (correct for binary CKC/license).
  // OVERRIDE via parseLicenseResponse for base64, JSON-wrapped, or other formats.
  const license = config.parseLicenseResponse
    ? await config.parseLicenseResponse(rawBody)
    : rawBody;

  await session.update(license);
};
```

### Step 5 — ClearKey Support

ClearKey (`org.w3.clearkey`) is the only EME system that works without a license server: keys can be embedded inline in the MPD via `<cenc:default_KID>` and a complementary key mapping.

Support declarative ClearKey via `drmConfig`:
```js
player.drmConfig = {
  'org.w3.clearkey': {
    keys: { '<keyId-hex>': '<key-hex>' }  // synthetic license, no server needed
  }
};
```

In `#onKeyMessage`, if key system is ClearKey and no `serverUrl` is set but `keys` map is present, build a synthetic JSON license:
```json
{"keys":[{"kty":"oct","k":"<base64url-key>","kid":"<base64url-keyId>"}],"type":"temporary"}
```

### Step 6 — Key Rotation Handling

Key rotation (common in live streams with 60-second key periods) is handled automatically by the `encrypted` event flow: each new key period generates a new init data value → new EME session → new license request. No special handling needed beyond correct deduplication.

To avoid session leaks on long-running live streams, add a session eviction policy: close sessions whose expiration time has passed (check `session.expiration` on each pump tick).

### Step 7 — Events and Error Surface

New custom events dispatched on `videl-player`. Per LEMMA's **CustomEvent contract** — events follow the existing `videl:` namespace pattern:

| Event | Detail | When |
|---|---|---|
| `videl:drm:ready` | `{ keySystem: string }` | MediaKeys successfully created |
| `videl:drm:error` | `{ error: Error }` | Key system access failed, license fetch failed, or `update()` rejected |
| `videl:drm:session-created` | `{ sessionId: string }` | New EME session created |

All events use `{ bubbles: true, composed: true }` per the existing event pattern.

### Step 8 — MPD Parser: `@ref`/`@refId` Resolution

**Corner case from spec §5.2.3.5 (ed.5)**:

```xml
<MPD>
  <ContentProtection refId="cpA" schemeIdUri="urn:uuid:widevine-guid" value="cenc">
    <cenc:pssh>...</cenc:pssh>
  </ContentProtection>
  <Period>
    <AdaptationSet>
      <ContentProtection ref="cpA"/>  <!-- reference to MPD-level -->
```

In `buildPresentation()`:
1. Build a `Map<string, Element>` of MPD-level `ContentProtection` elements keyed by `@refId`.
2. Pass this map down through `buildPeriod` → `buildAdaptationSet`.
3. In the CP collection step, if a CP element has `@ref`, look up the target and merge (referenced element's attributes/children supplement the referencing element).

### Step 9 — `cbcs` Scheme Support

`cbcs` (used in HLS-compatible CMAF content) requires:
- `initDataTypes: ['cenc']` still works for most browsers
- `MediaKeySystemConfiguration.videoCapabilities[].encryptionScheme = 'cbcs'` (where supported)

In key system config construction, detect `cbcs` from parsed protection data and set `encryptionScheme`:
```ts
const encryptionScheme = entries.some(e => e.value === 'cbcs') ? 'cbcs' : 'cenc';
videoCapabilities: [{ contentType, encryptionScheme }]
```

## FairPlay Summary

FairPlay **is** supported by this plan, with these additions over the Widevine/PlayReady path:

| What | Where | Notes |
|---|---|---|
| FairPlay UUID → key system ID mapping | `videl-player.ts` static constants | UUID `94ce86fb-...` → `com.apple.fps` |
| `initDataTypes: ['skd']` in `requestMediaKeySystemAccess` config | `#setupDrm` | Only when FairPlay is the active system |
| Certificate fetch + `setServerCertificate()` | `#setupDrm` (after `setMediaKeys`) | Blocking prerequisite before any session |
| `#fairPlayCert` stored on player | `videl-player.ts` | Needed in every `initDataTransform` call |
| `initDataTransform` hook on `DrmSystemConfig` | New field | Provider-specific key ID extraction; default impl included |
| Default envelope builder `#defaultFairPlayTransform` | `videl-player.ts` | `[initData][contentId][cert]` format per Apple spec |
| `Content-Type: application/octet-stream` default header | `#onKeyMessage` | Required by FairPlay license servers; harmless for others |
| `parseLicenseResponse` usage | Consumer code | Base64/JSON CKC decode for providers that wrap the response |

Consumer usage examples:

```js
// FairPlay with base64-encoded CKC response (common with Axinom, EZDRM, etc.)
player.drmConfig = {
  'com.apple.fps': {
    serverUrl: 'https://license.example.com/fps',
    certificateUrl: 'https://cdn.example.com/fps-cert.cer',
    // Override contentId extraction when provider doesn't use skd:// scheme
    initDataTransform: (initData, type, cert) => myTransform(initData, cert),
    // Provider returns base64-encoded CKC in response body
    parseLicenseResponse: async (body) => {
      const b64 = new TextDecoder().decode(body);
      return Uint8Array.from(atob(b64), c => c.charCodeAt(0)).buffer;
    }
  }
};

// Widevine with JSON-wrapped license (some custom proxy servers)
player.drmConfig = {
  'com.widevine.alpha': {
    serverUrl: 'https://license.example.com/widevine',
    parseLicenseResponse: async (body) => {
      const json = JSON.parse(new TextDecoder().decode(body));
      return Uint8Array.from(atob(json.license), c => c.charCodeAt(0)).buffer;
    }
  }
};
```
