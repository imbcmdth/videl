# Implementation Plan: Lazy Segment Generation + Sidx Support

**Date:** 2026-06-07  
**Status:** Ready for implementation  

---

## Overview

This plan describes the architectural shift that moves **segment enumeration** out of the MPD parser and into `VidelRepresentation`. The parser retains responsibility for building the element hierarchy and stamping manifest declarations onto representation elements as attributes. `VidelRepresentation` becomes the owner of segment lifecycle: it populates its own `<videl-segment>` children at activation time by interpreting those attributes.

The exception is **SegmentList**, which is a direct 1:1 XML-to-DOM transformation with no computation involved — it stays in the parser.

## Changes to `mpd-parser.ts`

### What the parser stamps instead of building segments

For **SegmentTemplate**, the parser stamps the raw template data as attributes on the representation element:

| New attribute | Source |
|---------------|--------|
| `segment-template-media` | `SegmentTemplate/@media` |
| `segment-template-timescale` | `SegmentTemplate/@timescale` (default 1) |
| `segment-template-start-number` | `SegmentTemplate/@startNumber` (default 1) |
| `segment-template-duration` | `SegmentTemplate/@duration` (if present) |
| `segment-template-timeline` | JSON-serialized `[{t, d, r}]` from `<SegmentTimeline><S>` elements |

The existing `timestamp-offset` attribute (derived from `@presentationTimeOffset`) stays — it is still calculated at parse time, which is correct since it depends on `periodStart` and the PTO.

The existing `initialization-url` and `initialization-byte-range` attributes also stay — the init segment is a manifest-level declaration and not subject to lazy enumeration.

For **SegmentBase**, the parser stamps:

| New attribute | Source |
|---------------|--------|
| `segment-base-url` | Resolved base URL for the representation |
| `segment-base-index-range` | `SegmentBase/@indexRange` |

The `Initialization` child (init URL / range) still stamps `initialization-url` and `initialization-byte-range` as before.

For **SegmentList**, nothing changes — the parser continues creating `<videl-segment>` elements directly (1:1 XML mapping, no computation, no benefit to deferring).

### Functions to remove from the parser

- `buildFromTimeline` 
- `buildFromNumber`
- `buildSegmentBase`
- The `appendSegment` helper and all call sites for SegmentTemplate/SegmentBase paths

### Functions that remain in the parser

- `buildSegmentList` — unchanged
- `readSegTemplate` / `mergeSegTemplate` — still needed to resolve inheritance; the merged result is now stamped as attributes instead of being used to build segments
- All `buildPresentation` / `buildPeriod` / `buildAdaptationSet` / `buildRepresentation` structure builders
- All URL resolution helpers

## Changes to `VidelRepresentation`

### New reactive properties

```ts
// SegmentTemplate fields (from new parser attributes)
segmentTemplateMedia:      string  // attribute: 'segment-template-media'
segmentTemplateTimescale:  number  // attribute: 'segment-template-timescale'
segmentTemplateStartNum:   number  // attribute: 'segment-template-start-number'
segmentTemplateDuration:   number  // attribute: 'segment-template-duration'
segmentTemplateTimeline:   string  // attribute: 'segment-template-timeline' (JSON)

// SegmentBase fields
segmentBaseUrl:            string  // attribute: 'segment-base-url'
segmentBaseIndexRange:     string  // attribute: 'segment-base-index-range'
```

### Segment population trigger

Add a private method `#populateSegments()` that:

1. Detects which addressing mode is in use (SegmentTemplate vs SegmentBase vs SegmentList) based on which attributes are present.
2. For **SegmentTemplate**: runs the existing `buildFromTimeline` / `buildFromNumber` logic (moved here from the parser) to append `<videl-segment>` children.
3. For **SegmentBase without sidx**: appends a single `<videl-segment>` pointing at `segment-base-url` with no byte range.
4. For **SegmentBase with sidx** (`segment-base-index-range` is present): fetches the sidx byte range, parses the ISOBMFF `sidx` box, and appends one `<videl-segment>` per entry with accurate byte ranges and start times.
5. For **SegmentList**: no-op — children already exist from the parser.

`#populateSegments()` is called from `attributeChangedCallback` when `videl-state` transitions to `"next"` or `"active"` and no `<videl-segment>` children are present yet. For a `"next"→"active"` transition, children will already be present from the `"next"` phase so no re-population is needed.

### Segment extension during pump ticks (live / dynamic manifests)

For live streams with a time-based SegmentTemplate (`$Number$` or `$Time$`), segment availability is a deterministic function of wall clock time — no manifest refresh is required to know that a new segment exists:

```
currentSegmentNumber = startNumber + floor((Date.now()/1000 - availabilityStartTime) / segmentDurationSec)
```

`videlUpdate()` already runs every 250 ms (configurable via `tick-ms`), which is far more granular than any `minimumUpdatePeriod`. This makes the pump tick the correct place to check for and append new `<videl-segment>` children.

**No separate `videlRefreshSegments()` public method is needed.** Instead, `videlUpdate()` gains a call to a private `#extendLiveSegments()` helper that:
1. Is a no-op for VOD representations (no `live` attribute).
2. Computes the segment number/time that should be available right now based on `Date.now()` and the stamped template attributes.
3. Compares against the last `<videl-segment>` child's declared `start-time` + `duration`.
4. Appends new `<videl-segment>` elements for any segments that have become available since the last tick.
5. Never touches existing children — append-only.

`PlayerState` does not need to carry wall clock time; the representation calls `Date.now()` directly since live segment availability is a real-time concern, not a playback-position concern.

This also means `minimumUpdatePeriod` is decoupled from segment availability. A manifest refresh (when implemented) updates structural metadata — `availabilityStartTime`, `timeShiftBufferDepth`, `startNumber` changes — but the pump continues fetching newly available segments at full tick rate regardless.

### `isFullyFetched` guard for live

`isFullyFetched` currently checks whether the last `<videl-segment>` child is in `#fetchedSegments`. For live representations there is no stable "last" segment. The guard should additionally check a `live` attribute (set by the parser when `MPD@type="dynamic"`) and return `false` unconditionally for live representations.

## Sidx Parsing Detail

### What sidx is

`sidx` (`SegmentIndexBox`, ISO 14496-12 §8.16.3) is an ISOBMFF box embedded in the media file that maps a single contiguous byte range into an ordered list of independently-decodable subsegments. For DASH `SegmentBase` streams it sits at the byte range declared in `SegmentBase/@indexRange`. The player fetches only those bytes (a single ranged GET), parses the box, and gains a complete segment index without downloading any media data.

### Wire format

All multi-byte integers are big-endian. The box begins with the standard 8-byte ISOBMFF header (4-byte size + 4-byte fourcc `sidx`), then:

```
// FullBox header
uint8   version          // 0 or 1
uint24  flags            // always 0

// Box body
uint32  reference_ID     // track_ID of the indexed track (informational)
uint32  timescale        // ticks per second for all time fields in this box

// version == 0: 32-bit times; version == 1: 64-bit times
if version == 0:
  uint32  earliest_presentation_time  // PTS of first sample in ticks
  uint32  first_offset                // byte distance from end of sidx box to first subsegment
else:
  uint64  earliest_presentation_time
  uint64  first_offset

uint16  reserved         // = 0
uint16  reference_count  // number of entries that follow

// repeated reference_count times:
  uint1   reference_type  // 0 = media subsegment, 1 = nested sidx (daisy-chain)
  uint31  referenced_size // byte length of this subsegment (excludes nothing)
  uint32  subsegment_duration  // duration of this subsegment in timescale ticks
  uint1   starts_with_SAP     // 1 if subsegment starts with a sync sample
  uint3   SAP_type            // SAP type (1-6); 0 if starts_with_SAP == 0
  uint28  SAP_delta_time      // ignored for our purposes
```

Total minimum size: 8 (ISOBMFF header) + 4 (FullBox header) + 4 (ref_ID) + 4 (timescale) + 8 (times v0) + 4 (reserved+count) = **32 bytes** before any entries. Version 1 adds 8 extra bytes for the 64-bit times.

Each entry is exactly **12 bytes**.

### Byte offset arithmetic

The byte offset of each subsegment is calculated by walking the entries and accumulating `referenced_size`. The base byte address for entry 0 is:

```
subsegment_0_start = (byte offset of sidx box end) + first_offset
```

where "sidx box end" is `indexRange.end + 1` (the byte immediately after the last byte of the fetched range). Each subsequent subsegment starts immediately after the previous one:

```
subsegment_N_start = subsegment_0_start + sum(referenced_size[0..N-1])
subsegment_N_end   = subsegment_N_start + referenced_size[N] - 1
byte-range attribute = "${subsegment_N_start}-${subsegment_N_end}"
```

### Presentation time arithmetic

```
segment_0_startTime = earliest_presentation_time / timescale  (media-timeline seconds)
segment_N_startTime = segment_0_startTime + sum(subsegment_duration[0..N-1]) / timescale
segment_N_duration  = subsegment_duration[N] / timescale
```

These are **media-timeline values** — they must be adjusted by `timestampOffset` (already stamped on the representation by the parser) before being used as MSE presentation times, consistent with how all other segment `start-time` attributes are treated.

### Nested sidx (`reference_type == 1`)

Some streams use daisy-chained sidx boxes (a top-level sidx whose entries point to further sidx boxes rather than media). Each such entry has `reference_type == 1`. For V1:

- **Skip them.** Log a `videl:trace` event at `category: 'fetch', action: 'sidx-nested-skipped'` and continue to the next entry.
- Do not attempt recursive sidx fetching — this is an edge case found mainly in very large VOD assets or broadcast-originated content. It can be implemented in a follow-up.

### What to ignore

- `reference_ID` — informational only; not used
- `SAP_type` and `SAP_delta_time` — used for trick play / random access optimisation; out of scope
- `starts_with_SAP` — useful for ABR switching heuristics later; record but don't act on it now

### `parseSidx` function signature and contract

Location: `src/lib/mp4/sidx-parser.ts` (co-located with other box utilities, not in `src/parser/`)

```ts
export interface SidxEntry {
  /** Absolute byte range string for the Range header, e.g. "1024-8191" */
  byteRange:  string;
  /** Presentation start time in seconds (media timeline, before timestampOffset) */
  startTime:  number;
  /** Duration in seconds */
  duration:   number;
}

/**
 * Parse a sidx (SegmentIndexBox) from `buffer`.
 *
 * @param buffer         ArrayBuffer containing ONLY the fetched sidx bytes
 *                       (i.e. the response body for the indexRange GET).
 * @param sidxEndByte    The absolute byte offset in the media file where the
 *                       fetched range ends (= indexRange end + 1). Used to
 *                       compute subsegment byte addresses.
 * @returns              Ordered array of subsegments. Nested-sidx entries
 *                       (reference_type == 1) are omitted with a console.warn.
 *                       Returns [] if the buffer does not contain a valid sidx box.
 */
export function parseSidx(
  buffer:      ArrayBuffer,
  sidxEndByte: number
): SidxEntry[]
```

The function uses the existing `findBox` and `readUint32BE` / `readUint64BE` helpers from `src/lib/mp4/box-utils.ts`. No new low-level primitives are needed.

### Integration in `VidelRepresentation`

`#fetchAndParseSidx()` is called from `#populateSegments()` when `segment-base-index-range` is present:

```
1. Parse the byte range from the attribute: "start-end"
2. Fetch GET segment-base-url with header Range: bytes=start-end
   (reuse the existing AbortController pattern from #startInit)
3. Await response.arrayBuffer()
4. Call parseSidx(buffer, end + 1)
5. For each SidxEntry, createElement('videl-segment') and stamp:
     url        = segment-base-url   (same URL, different byte range)
     byte-range = entry.byteRange
     start-time = entry.startTime
     duration   = entry.duration
6. appendChild each to this (the representation element)
7. Emit videl:trace { category: 'fetch', action: 'sidx-parsed',
     data: { url, entryCount, firstStartTime, totalDuration } }
```

If the fetch fails or `parseSidx` returns an empty array, emit `videl:segment:error` and do not append any children — the representation will remain without segments and the pump will stall, which is the correct observable failure mode (consistent with a failed init segment fetch).

### Test fixture

A minimal valid sidx buffer (version 0, 3 entries) should be added to `test/fixtures/` as a binary file and used for the `parseSidx` unit test. The fixture can be constructed programmatically in the test setup using `DataView` writes rather than requiring a real media file.

## Shifted Architecture Summary

```
Before:
  parseMpd()
    └─ buildRepresentation()
         └─ buildSegments()           ← creates all <videl-segment> children
              ├─ buildFromTimeline()
              ├─ buildFromNumber()
              └─ buildSegmentBase()

After:
  parseMpd()
    └─ buildRepresentation()          ← stamps template/base attrs, no segment creation
         └─ buildSegmentList()        ← ONLY for SegmentList (unchanged)

  VidelRepresentation.attributeChangedCallback('videl-state' → 'next'|'active')
    └─ #populateSegments()
         ├─ #buildFromTemplate()      ← moved from parser
         ├─ #buildFromSegmentBase()   ← moved from parser
         └─ #fetchAndParseSidx()      ← new
```

## Impact on Live / `type="dynamic"` Support

This is the key architectural benefit. For live DASH with a time-based SegmentTemplate:

1. **The pump tick owns segment discovery.** `#extendLiveSegments()` is called on every `videlUpdate()`. New segments append themselves to the `<videl-segment>` list the moment wall clock time indicates they are available — independent of any manifest refresh cycle. A `minimumUpdatePeriod` of 60 seconds does not stall segment fetching for 2-second segments.

2. **`minimumUpdatePeriod` only covers structural metadata.** A manifest refresh updates `availabilityStartTime`, `timeShiftBufferDepth`, `startNumber` corrections, and any period/adaptation-set changes. Crucially it does NOT need to rebuild the element tree or reset buffer state — it updates attributes on the existing representation elements, and the pump naturally incorporates those changes on its next tick.

3. **Segment expiry** (sliding availability window): the representation can prune `<videl-segment>` elements that have fallen outside the `timeShiftBufferDepth` window and whose content is no longer on the server — as long as those segments are not in `#fetchedSegments` (already buffered) and not currently `videl-state="active"` or `"next"`.

4. **`isFullyFetched` stays `false`** for live representations — the period never signals "done" so the player never advances to the next period.

### What `type="dynamic"` will require beyond this work

- `<videl-presentation>` polls the MPD URL on a `minimumUpdatePeriod` interval
- A targeted attribute-update path (not a full tree rebuild) to update `segment-template-*` and `availability-start-time` on existing representation elements after each manifest refresh
- `MediaSource.setLiveSeekableRange()` called periodically from `videl-player` (already noted in LEMMA.md)
- `isFullyFetched` returning `false` unconditionally for `live` representations (described above)
- `availability-start-time` stamped on `<videl-representation>` at parse time (propagated from `MPD@availabilityStartTime` → Period → Representation) so `#extendLiveSegments()` can compute available segment numbers directly from the DOM

This plan lays the groundwork for all of those without implementing them. The critical point is that **segment discovery at tick time is already the right model** — `type="dynamic"` support layers on top rather than requiring a different architecture.

## LEMMA Addition

The following binding decision should be recorded in `LEMMA.md` as part of this work:

> **DOM-first data model:** Prefer stamping data as attributes on custom elements over storing it in internal JavaScript variables or objects. The DOM tree is the authoritative state of the player — attributes represent the externally observable, inspectable, and diffable data model. Internal JavaScript fields (`#field`) represent only transient tracking state that cannot be meaningfully observed from outside the element (e.g. `#fetchedSegments`, `#timelineDrift`, `#initController`). The key test: if an internal field could be fully reconstructed by re-reading the element's attributes and re-running its activation logic from scratch, it belongs as an attribute, not a field.

This principle already governs how segment template data, timestamp offsets, and codec strings flow from parser to representation. It should be made explicit so future work (live metadata, sidx byte ranges, `availabilityStartTime`, `timeShiftBufferDepth`) follows the same pattern rather than accumulating internal state.

## Test Considerations

- Existing parser tests that assert `<videl-segment>` children on parsed representations using SegmentTemplate/SegmentBase will need to be updated — those children are no longer present after parsing; they appear after activation.
- New unit tests for `parseSidx` in isolation: construct a known-good sidx buffer programmatically using `DataView` writes (version 0 and version 1 variants), assert the returned `SidxEntry[]` values for byte ranges, start times, and durations. Test the nested-sidx skip path. Test an empty / malformed buffer returns `[]`.
- New representation activation tests: SegmentTemplate → expect segments populated after `videl-state="next"`.
- `#extendLiveSegments()` tests: stub `Date.now()`, verify that new `<videl-segment>` elements are appended on successive `videlUpdate()` calls as wall clock time advances past each segment boundary. Verify no existing children are disturbed.
