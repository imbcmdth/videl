# Plan: Promote Internal State to DOM Attributes + Attribute Naming Convention

## Background

LEMMA.md (2026-06-07) establishes the DOM-first data model principle:

> Prefer stamping data as attributes on custom elements over storing it in internal JavaScript variables or objects. The key test: if an internal field could be fully reconstructed by re-reading the element's attributes and re-running its activation logic from scratch, it belongs as an attribute, not a field.

This plan has two parts:
1. **Attribute naming convention** — a new LEMMA rule clarifying which attributes use the `videl-` prefix and which don't, plus a full audit of existing attributes that violate this rule.
2. **Internal → attribute promotions** — private fields that should become DOM attributes per the DOM-first principle.

---

## Part 1: Attribute Naming Convention

### The Rule (to be added to LEMMA.md)

Three categories, three treatments:

| Origin | Prefix | Examples |
|--------|--------|---------|
| **DASH manifest** — stamped directly from MPD XML or computed from MPD values | none | `bandwidth`, `codecs`, `duration`, `start`, `lang`, `live`, `timestamp-offset` |
| **User configuration** — set by the consumer to configure behavior | none | `src`, `debug`, `tick-ms`, `buffer-ahead`, `abr-safety-factor`, `forced-rep` |
| **Internal state** — set by videl processes to reflect computed or lifecycle state | `videl-` | `videl-state`, `videl-done`, `videl-populated`, `videl-bandwidth` |

The key diagnostic question: **"Who writes this attribute?"**
- MPD parser writes it → no prefix
- Consumer writes it → no prefix
- videl element writes it to record its own state → `videl-` prefix

`slot` is a built-in HTML attribute and is exempt.

---

### Existing Attributes That Violate This Rule (Require Renaming)

These are all **internally-written state** attributes currently lacking the `videl-` prefix:

#### 1. `generated` on `<videl-presentation>` → `videl-generated`

Set by `videl-player` when it creates a presentation from the `src` attribute in single-stream mode. Used in CSS to collapse the generated presentation card. No consumer sets it.

**Files affected:** `videl-player.ts` (set site), `videl-presentation.ts` (CSS `:host([generated])`).

#### 2. `no-playlist` on `<videl-player>` → `videl-no-playlist`

Set by `videl-player.#refreshPlaylistChrome()` when fewer than 2 presentations exist. Used in CSS to collapse the playlist column. No consumer sets it.

**Files affected:** `videl-player.ts` (set/remove + CSS `:host([no-playlist])`).

#### 3. `user-inactive` on `<videl-player>` and `<videl-presentation>` → `videl-user-inactive`

Set by `videl-player.#onInactivityTimeout()`. Propagated to the active presentation. Used in CSS to hide controls. No consumer sets it.

**Files affected:** `videl-player.ts` (set/remove + CSS `:host([user-inactive])`), `videl-presentation.ts` (CSS `:host([user-inactive])`).

#### 4. `menu-open` on `<videl-period>` → `videl-menu-open`

Set by `videl-presentation` on the active period to reveal the menu popup. No consumer sets it — it is driven entirely by the presentation's control bar button clicks. The `menu-open` property on `videl-presentation` is currently `attribute: false` (internal only); when promoted to an attribute (see Part 2) it also gets the `videl-menu-open` name.

**Files affected:** `videl-period.ts` (Lit property declaration, CSS selectors `:host([menu-open="audio"])` etc., `attributeChangedCallback`, `render()`), `videl-presentation.ts` (`#toggleMenu`, `#closeMenu`, `videlUpdate`, CSS).

#### 5. `pinned` on `<videl-representation>` → `videl-pinned`

Set by `videl-adaptation-set.#updatePinnedAttrs()` to mark which representation is ABR-pinned. No consumer sets it — it reflects the adaptation-set's internal ABR state.

**Files affected:** `videl-adaptation-set.ts` (set/remove in `#updatePinnedAttrs`), `videl-representation.ts` (CSS `:host([pinned])`, `render()` pin icon).

## Part 2: Internal State → Attribute Promotions

### 1. `<videl-representation>` — `#initAppended` → `videl-init-appended`

**Current state:** `#initAppended: boolean = false` — flips to `true` once the init segment has been appended to the SourceBuffer.

**Why it belongs on the DOM:** The debug render already shows `init=<em>${this.#initAppended ? 'done' : 'pending'}</em>`. External observers (test harnesses, CSS loading indicators) cannot inspect init state today. Binary, set once, never cleared until deactivation.

**Proposed attribute:** `videl-init-appended` (boolean presence attribute).

**Impact:**
- `videlUpdate()` gate: `if (!this.#initAppended)` → `if (!this.hasAttribute('videl-init-appended'))`.
- `#startInit` chain: `this.setAttribute('videl-init-appended', '')` after the append completes (where `this.#initAppended = true` currently is).
- `attributeChangedCallback` for `videl-state=null`: `this.removeAttribute('videl-init-appended')` (replaces `this.#initAppended = false`).
- `sourceBuffer` setter: `this.removeAttribute('videl-init-appended')` (replaces `this.#initAppended = false`).
- `#initAppended` field is removed entirely.
- Debug render: change field reference to `this.hasAttribute('videl-init-appended')`.

---

### 2. `<videl-presentation>` — `#populated` → `videl-populated`

**Current state:** `#populated: boolean = false` — flips to `true` once the MPD has been fetched and parsed.

**Why it belongs on the DOM:** The debug render shows `populated=<em>${this.#populated}</em>`. External code waiting for `videlPopulate()` to complete, CSS loading states, and test harnesses cannot observe this today.

**Proposed attribute:** `videl-populated` (boolean presence attribute).

**Impact:**
- `#populate()` success path: `this.setAttribute('videl-populated', '')` (replaces `this.#populated = true`).
- `#ensurePopulated()` early-return: `if (this.hasAttribute('videl-populated'))` (replaces `if (this.#populated)`).
- `#populated` field is removed entirely.
- Debug render: change field reference to `this.hasAttribute('videl-populated')`.

---

### 3. `<videl-period>` — `#doneEmitted` → `videl-done`

**Current state:** `#doneEmitted: boolean = false` — guards against firing `videl:done` more than once per activation.

**Why it belongs on the DOM:** Period completion is a significant playback event. External code (ad-insertion, test assertions about period advancement) cannot determine from the DOM whether a period has already completed.

**Proposed attribute:** `videl-done` (boolean presence attribute, set when `videl:done` fires, removed when `videl-state` is cleared).

**Impact:**
- `videlUpdate()` guard: `if (!this.#doneEmitted && this.duration !== null)` → `if (!this.hasAttribute('videl-done') && this.duration !== null)`.
- On fire: `this.setAttribute('videl-done', '')` before (or alongside) dispatching the event.
- `attributeChangedCallback` for `videl-state=null`: `this.removeAttribute('videl-done')` (replaces `this.#doneEmitted = false`).
- `#doneEmitted` field is removed entirely.

---

### 4. `<videl-presentation>` — pump-driven UI properties → `videl-*` attributes

**Current state:** Eight Lit reactive properties are declared `attribute: false` or with no attribute binding:

```typescript
currentTime   = 0;    // attribute: false
paused        = true; // attribute: false
volume        = 1;    // attribute: false
muted         = false;// attribute: false
seekableStart = 0;    // attribute: false
seekableEnd   = 0;    // attribute: false
menuOpen      = null; // attribute: false
fullscreen    = false;// attribute: false
```

**Why they belong on the DOM:** These are pump-stamped values representing live playback state. External tools (`media-chrome` compatibility, CSS `[videl-paused]` selectors, automation) cannot observe them today. The `videl-` prefix is required because these are written by internal videl processes, not sourced from the DASH manifest.

**Proposed attributes:**
| JS Property | DOM Attribute | Type |
|-------------|---------------|------|
| `currentTime` | `videl-current-time` | Number (written each tick) |
| `paused` | `videl-paused` | Boolean presence |
| `volume` | `videl-volume` | Number |
| `muted` | `videl-muted` | Boolean presence |
| `seekableStart` | `videl-seekable-start` | Number |
| `seekableEnd` | `videl-seekable-end` | Number |
| `menuOpen` | `videl-menu-open` | String or absent |
| `fullscreen` | `videl-fullscreen` | Boolean presence |

**Impact:**
- In Lit `properties`, change `attribute: false` to `attribute: 'videl-current-time'`, etc.; add `reflect: true`.
- For `paused`, `muted`, `fullscreen` (booleans): Lit's `reflect: true` on a Boolean type correctly adds/removes the attribute.
- `videlUpdate()` still assigns the JS properties — Lit syncs to attributes automatically via reflection.
- The manual `requestUpdate()` calls in `videlUpdate`, `#closeMenu`, `#toggleMenu`, and `#onFullscreenChange` must be kept because of the `useDefineForClassFields` shadowing issue noted in the existing code comment.

**Note on `videl-menu-open` synchronization:** Both `videl-presentation` (new `videl-menu-open` attribute) and `videl-period` (renamed from `menu-open`) use this attribute name. They are distinct elements; the presentation owns the menu state and stamps it onto the active period. This remains the same ownership model — only the attribute name changes on both.

---

### 5. `<videl-player>` — `#bandwidth` → `videl-bandwidth`

**Current state:** `#bandwidth: number = 1_000_000` — EWMA-estimated download bandwidth in bits per second.

**Why it belongs on the DOM:** Drives ABR decisions. External consumers (custom ABR overrides, monitoring dashboards) cannot observe it without event subscription.

**Proposed attribute:** `videl-bandwidth` (Number, written after each EWMA update, rounded to nearest integer bps).

**Impact:**
- After updating `#bandwidth` in `#onVidelDone`: `this.setAttribute('videl-bandwidth', String(Math.round(this.#bandwidth)))`.
- `#bandwidth` field stays (it is the fractional EWMA accumulator; the attribute mirrors the rounded value).
- `videl-bandwidth` is write-only from within the element; no `observedAttributes` change needed.

---

### 6. `<videl-adaptation-set>` — `#activeMimeAndCodecs` → `videl-active-codecs`

**Current state:** `#activeMimeAndCodecs: string | null = null` — the full MIME+codec string of the currently active representation.

**Why it belongs on the DOM:** External observers cannot tell what codec is currently decoding. Completes the observable codec state picture alongside the existing `mime-type` and `codecs` manifest attributes.

**Proposed attribute:** `videl-active-codecs` (String, set by `#performSwitch`, removed on deactivation).

**Impact:**
- `#performSwitch`: `this.setAttribute('videl-active-codecs', newMimeAndCodecs)` replaces `this.#activeMimeAndCodecs = newMimeAndCodecs`.
- `attributeChangedCallback` for `videl-state=null`: `this.removeAttribute('videl-active-codecs')` replaces `this.#activeMimeAndCodecs = null`.
- The `#performSwitch` codec-change guard: `this.#activeMimeAndCodecs` → `this.getAttribute('videl-active-codecs')`.
- `#activeMimeAndCodecs` field is removed entirely.

## Part 3: LEMMA.md Addition

Add the following entry dated 2026-06-07 to the Decisions section:

```
- **[2026-06-07] Attribute naming convention:** Three categories govern whether an attribute uses the `videl-` prefix.
  1. **DASH-manifest attributes** — stamped by the MPD parser from MPD XML values (including values derived or computed from MPD data such as `timestamp-offset`, `period-id`, `live`): **no prefix**. Match the DASH attribute name as closely as possible in kebab-case.
  2. **User-configuration attributes** — set by the element consumer to configure behavior (`src`, `debug`, `tick-ms`, `buffer-ahead`, `abr-safety-factor`, `forced-rep`, `playlist-collapsed`): **no prefix**.
  3. **Internal-state attributes** — written by videl processes to record lifecycle or computed state that consumers observe but do not set (`videl-state`, `videl-done`, `videl-populated`, `videl-init-appended`, `videl-bandwidth`, `videl-active-codecs`, `videl-generated`, `videl-no-playlist`, `videl-user-inactive`, `videl-menu-open`, `videl-pinned`, pump-state attrs on `videl-presentation`): **`videl-` prefix required**.
  The diagnostic question: *who writes this attribute?* Parser → no prefix. Consumer → no prefix. videl element writes it to reflect its own state → `videl-` prefix. The HTML built-in `slot` attribute is exempt from all three rules.
```

## Out of Scope

- `#fetchStats` on `videl-segment` — data already fully surfaced via the `videl:done` event payload; duplicating as attributes creates two sources of truth with no clear consumer benefit.
- `#lastCurrentTime` on `videl-period` — pure internal delta used for SourceBuffer trimming; no semantic meaning to external observers.
- `#lastState` on `videl-adaptation-set` — a `PlayerState` object; cannot be meaningfully represented as attributes.
